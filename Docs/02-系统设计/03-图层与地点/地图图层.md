## 目标

定义正六边形格子上多层内容的组成方式、各图层的堆叠规则，以及**响应检测**如何在他人行动阶段触发**行为**。

## 范围

- **包含**：图层类型、默认可叠加规则、各图层堆叠逻辑、图层间影响规则与优先级、**响应检测**与行为触发、示例。

- **不包含**：影响规则的具体数值、完整规则表、行为通道内的具体数值、程序实现。

## 详细说明

### 基本原则

- 一个格子可同时含有多个图层的内容。

- 一般情况下，不同图层的内容可在同一格内叠加。

- 在此基础之上，存在图层之间的**影响逻辑**；各图层内部也有各自的重叠逻辑。

### 地形层与环境层的分工

两层的划分以**是否描述地面形状**为准，而非是否「自然形成」：

| 图层 | 职责 | 典型内容 | 变更特征 |
|------|------|----------|----------|
| **地形层** | 格子的**地面形状**与地貌类型：决定通过性、占格、桥梁/裂谷等空间规则 | 平原、丘陵、山地、河流、裂谷、沼泽等 | 每格一种，作为底层；通常随地图生成确定，较少因短期行为改变 |
| **环境层** | 格子上的**表层状况**与**大气/区域效果**：地面植被与覆盖、灾害天气、移动城市轨迹等 | 树林、焦土、弹坑；沙暴、暴雨、迷雾；移动城市轨迹等 | 可随时间或单位行为产生、变更或消退（存续时间可长可短） |

- **树林、密林**等植被覆盖属于**环境层**（表层状况），不属于地形层；同一格可同时有丘陵（地形）与密林（环境）。
- **天气类**效果（沙暴、暴雨、迷雾等）与环境层中的**地面表层**效果（焦土、弹坑等）同层叠放，各自类型自有重叠逻辑。
- 旧稿将密林列在地形层的写法**废止**；影响规则中凡「密林削弱视野」类条目，触发条件改为**环境层**。

### 地形类型清单

首版地形层类型固定为下列六种，均属**地面形状**（见 [地形层与环境层的分工](#地形层与环境层的分工)）。每格**恰好一种**，作为底层，不可与同层另一地形叠加。

**路径等效格数**：队伍移动结算时，进入或经过某地形格，按该类型的**路径等效格数**计入本回合路径消耗（与 [地图与移动 · 移动规则](../02-地图与世界/地图与移动.md#移动规则) 合成；具体公式 **待定**，sy-02）。

**参数权威来源（各图层 CSV）**：总览 [地图图层配置数据结构](../../03-程序设计/03-数据字典/地图图层配置数据结构.md) · 目录 [tables/README.md](../../03-程序设计/03-数据字典/tables/README.md)

**本层（L1 地形）**：[L1_terrain_defs.csv](../../03-程序设计/03-数据字典/tables/L1_terrain_defs.csv)

每格地形在运行时挂载 **hex `AbilitySystemComponent`**：导入管线按 profile **列值**生成 GE/GA，不在程序里 `if (terrain==X)` 写死。

**可建造设施**分两层：

1. **基础档位** `build_mode`（三档）
2. **允许设施白名单** `build_allow_facility_ids`：指定 `facility_id`，**优先级高于**基础档位

| 档位 | CSV 值 | 含义 |
|------|--------|------|
| **全部允许** | `all` | 占格类、辅助类、地形类均可（仍受 `facility_type` SO 的 `rule_guard` 约束） |
| **仅辅助类** | `auxiliary_only` | 仅辅助类；生产/仓储与地形类不可 |
| **全部禁止** | `none` | 阻断一般 `TryBuild`；**白名单**中的类型除外 |

白名单目录：[build_grant_tags.csv](../../03-程序设计/03-数据字典/tables/_program/build_grant_tags.csv) · 资源点：[L3_resource_defs.csv](../../03-程序设计/03-数据字典/tables/L3_resource_defs.csv)

**结算顺序**（`FacilityBuildRuleEvaluator` / `TryBuild`）：

1. 目标格是否持有 `Allow.Build.{facility_id}` → **允许**（跳过基础档位封锁）
2. 否则按 `build_mode` 判定类别是否可建
3. 再校验 `facility_type_config.build_rule_guard_json`

| 值 | 中文 | 含义 |
|----|------|------|
| `passable` | **可通过** | 可正常经行、占格 |
| `crossable` | **可跨越** | 可部分格通过；**禁止**同时让所有连接城区均落在此类格上 |
| `blocked` | **不可通过** | 不可经行、不可占格 |

| 类型 | 单位路径 | 单位可进入 | 基础档位 | 允许设施白名单 | 移动城市 | profile |
|------|:--------:|:----------:|:--------:|----------------|----------|---------|
| **平原** | 1 | 是 | 全部允许 | — | 可通过 | `terrain_profile_plain` |
| **丘陵** | 2 | 是 | 仅辅助类 | — | 可通过 | `terrain_profile_hill` |
| **山地** | 5（隧道有效时→1） | 是 | 全部禁止（隧道有效时→仅辅助类） | `tunnel` | 可通过 | `terrain_profile_mountain` |
| **河流** | 4 | 否 | 全部禁止 | `bridge` | **可跨越** | `terrain_profile_river` |
| **沼泽** | 3 | 是 | 全部禁止 | — | 可通过 | `terrain_profile_swamp` |
| **裂谷** | 1 | 否 | 全部禁止 | `bridge` | **可跨越** | `terrain_profile_rift` |

**资源点**（资源层，与地形正交）：村镇 / 矿藏 / 果地 / 遗迹 各自挂载对应**采集设施**允许规则（如 `allow_mine_extraction`），同样覆盖该格地形基础档位。

#### 河流与裂谷（阻断地形）及桥梁

**河流**对单位路径等效 **4** 格；**通行封锁、桥梁前提、移动城市跨行**与裂谷相同。桥梁通过 `allow_bridge` 授予 `Allow.Build.bridge`，再走通用 `TryBuild`。

| 项 | 规则 |
|----|------|
| **一般单位通行** | profile 基线阻断进入；**桥梁有效时**由 bridge 设施配置恢复（`when_valid_override_unit_enter_allowed` + 跨格连通） |
| **路径等效（单位）** | 河流 **4**；裂谷 **1**（`GE_Terrain_PathCost_4` / `GE_Terrain_PathCost_1`） |
| **桥梁前提** | 阻断地形**两端**各存在至少一格**非该阻断类型**地形（河流两端非河流、裂谷两端非裂谷）；两端之间以**直线**连接架设桥梁 |
| **桥梁效果** | 建成后，一般单位可经桥梁跨越（占用与耐久细则 **待定**，sy-10） |
| **移动城市** | `mobile_city_pass=crossable`：可跨越河流/裂谷；**禁止**同时让所有连接城区均落在此格 |

#### 沼泽

| 项 | 规则 |
|----|------|
| **路径等效（单位）** | **3**（`GE_Terrain_PathCost_3`） |
| **可建造设施** | **全部禁止**（`build_mode=none`） |
| **改造** | — |

#### 山地与隧道

| 项 | 规则 |
|----|------|
| **建造** | 白名单含 `tunnel` → `Allow.Build.tunnel` |
| **有效时效果** | **地形类设施基本能力**（[L5_facility_defs.csv](../../03-程序设计/03-数据字典/tables/L5_facility_defs.csv)）：路径 **1**、建造档位 **仅辅助类**；击毁回退 |
| **基线** | profile：`unit_path_cost_equiv=5`，`build_mode=none` |

- **树林、焦土、弹坑**等不属于本表，见 [环境层](#地形层与环境层的分工)。
- **荒地、暗渊**为区域或玩法光带概念（见 [胜利条件](../01-核心体验/胜利条件.md)、设定 [荒地](../../04-设定/03-地点与场景/荒地.md)），**不是**地形层类型枚举项。
- **遗迹**属于 [资源层](#资源层与采集设施)，与地形层正交。
- 旧称 **裂隙** 与 **裂谷** 同指本表 **裂谷** 类型（2026-07-06 起统一用 **裂谷**）。

### 图层间影响逻辑

图层之间默认可堆叠；当多种图层内容同时存在时，按**影响规则**修正互动结果。

**优先级（必须）**

- 每一条影响规则**必须**绑定独立的优先级。

- 结算时按优先级顺序依次判定；优先级数值越小，越先结算。

- 即使两条规则对调顺序不会改变最终结果（例如密林与丘陵对哨塔视野的叠加），每条规则仍须各自持有优先级，不得省略。

### 影响规则示例

| 优先级 | 触发条件 | 效果 |
|--------|----------|------|
| 10 | 地形层为**河流**或**裂谷** | 基础档位 **全部禁止**；**不可**由一般单位直接进入或停留 |
| 12 | 地形层为**沼泽** | 基础档位 **全部禁止** |
| 15 | 地形层为**山地** | 基础档位 **全部禁止**（隧道 / 允许设施除外） |
| 20 | 地形层为**河流**或**裂谷**，profile 含 `allow_bridge`，且满足桥梁几何 guard | 授予 `Allow.Build.bridge`；允许架设 **桥梁**；建成后一般单位可经桥梁跨越 |
| 22 | 资源层存在**矿藏 / 果地 / 遗迹 / 村镇**，且挂载 `build_allow_facility_ids` | 授予 `Allow.Build.{采集设施}`；**覆盖**该格地形基础档位 |
| 25 | 地形层为**山地**，设施层存在**有效隧道** | 由 `tunnel` 设施配置覆盖：路径 **1**、建造 **仅辅助类**（非地形 capability） |
| 30 | 设施层存在**哨塔**，环境层为**密林**（或等效的树林/植被覆盖） | 削弱哨塔视野 |
| 40 | 设施层存在**哨塔**，地形层为**丘陵** | 强化哨塔视野（程度待定） |
| 50 | 设施层存在**哨塔**，地形层为**山地** | 强化哨塔视野（程度高于丘陵，数值待定） |

说明：

- 优先级 10 与 20 同时判定时：先应用基础档位封锁，**允许设施**（`Allow.Build.*`）在 `TryBuild` 中优先放行桥梁。
- 优先级 22 与 10/12 同时判定时：资源点采集设施可建于沼泽等「全部禁止」地形格上。

- 优先级 30、40、50 同时作用于同一哨塔时，按序依次结算；具体叠加方式（累加 / 取极值等）待定，但各条规则的优先级不可合并省略。

### 图层一览

由下至上，典型叠放顺序如下：

| 图层 | 内容 | 堆叠规则 |
|------|------|----------|
| 地形层 | **地面形状**：平原、丘陵、山地、河流、沼泽、裂谷等（见 [地形层与环境层的分工](#地形层与环境层的分工)） | 每格一种，作为底层 |
| 环境层 | **表层状况与区域效果**：树林、焦土、弹坑；沙暴、暴雨、迷雾；移动城市轨迹等 | 各环境类型自有重叠逻辑 |
| 资源层 | 资源点（村镇、矿藏、果地、遗迹）及**储量**；**三级揭示**（`hidden` / `kind` / `amount`）；可建立对应**采集设施** | 不堆叠同类；多种资源点可并存；统计储量 |
| 建筑层 | **城区**（见 [建筑层与城区](#建筑层与城区)） | 不可堆叠 |
| 设施层 | **占格类**（地形 / 生产 / 仓储）与**辅助类**设施；详见 [设施层 · 设施分类](设施层.md#设施分类占格类--辅助类) | 堆叠规则受已有设施影响 |
| 物品层 | 可被单位拾取的物品 | 堆叠上限受其他图层影响 |
| 单位层 | 队伍单位 | 默认不可堆叠；部分设施/建筑可容纳多个单位 |

**数值策划入口**：各图层策划主表一行一类（[L1 地形](../../03-程序设计/03-数据字典/tables/L1_terrain_defs.csv) 已采用中文列名试点）。总览 [地图图层配置数据结构](../../03-程序设计/03-数据字典/地图图层配置数据结构.md) · 填写规范 [策划可读配置表规范](../../00-规范/策划可读配置表规范.md) · 文件清单 [tables/README.md](../../03-程序设计/03-数据字典/tables/README.md)。

### 资源层与采集设施

- **资源层**格含 **储量**、资源点类型，以及对玩家的 **`resource_reveal_level`**（**隐藏 → 揭示种类 → 揭示储量**，见 [单位类型与视野 · 资源点揭示等级](../06-单位与交战/单位类型与视野.md#资源点揭示等级)）。
- **占格类 · 生产类（储量转化）**建于资源点，通过 [工作](../07-玩法循环/工作.md) 将储量转化为物品层资产；类型包括 **征兵办**、**矿区**、**果园**、**能源站**（见 [设施层](设施层.md#占格类--三类细分)）。
- **遗迹**（资源层）：**前文明遗迹**，**不可重建**；储量产出能源，仅可建 **能源站** 等采集设施。
- **废墟**不是独立图层对象，而是**城区**的一种**状态**（见 [建筑层与城区](#建筑层与城区)）；与资源层 **遗迹** 的区分见 [荒野地点 · 遗迹与废墟](../04-资源与人口/荒野地点.md#遗迹与废墟术语强制区分)。

### 建筑层与城区

**建筑层存储的内容就是城区**——二者划等号，不另设与城区平行的「城市建筑类型」。

| 字段 | 说明 |
|------|------|
| **对象** | **城区**：规则见 [城市模块化](建筑层/README.md) |
| **类别** | **特殊城区** / **一般城区**（正交于状态） |
| **状态** | **正常** / **废墟** |
| **堆叠** | 每格至多一个城区实例 |

- **正常**：按类别提供能力；可承载人口（若配置允许）。
- **废墟**：被击毁或默认刷新的损毁城区；玩家须能**认出**其为废墟；**结构完整度 低于 20%** 强制转废墟；**无法工作**；**不可迁入**、已有居民**仅可迁出**；**须修复为正常**后机制上才与正常城区同等对待。
- **外部城市**、玩家**移动城市**占格：由一格或多格**城区**组成或挂载于建筑层；移动城市的城区连接关系另见 [城市模块化](建筑层/README.md) 与 [地图与移动](../02-地图与世界/地图与移动.md)。

### 图层与地点类型的关系

- **建筑层**上的城区、外部城市占格，与 [荒野地点 · 建筑层城区](../04-资源与人口/荒野地点.md#建筑层城区) 对应；玩家**移动城市**见 [城市模块化](建筑层/README.md)。
- **设施层**中的采集设施、驿站等，与 [探索与扩张](./探索与扩张.md) 中的建设行为对应。

- **资源层**与 [四种核心资源](../04-资源与人口/四种核心资源.md) 的产出来源相关。

### 行为通道

地图上的**行为**是实际结算单元；凡改变局面状态的执行，均经下列**行为通道**完成（见 [工作](../07-玩法循环/工作.md)、[交战系统](../06-单位与交战/交战系统.md)、[回合与行动表](../07-玩法循环/回合与行动表.md)）：

| 通道 | 说明 | 典型对象 |
|------|------|----------|
| **战斗** | 即时伤害、**减员**、**建筑损伤**、**设施耐久**损失；不走工作进度 | [交战系统](../06-单位与交战/交战系统.md) |
| **工作** | 工作进度推进；达阈值发放工作成果 | [工作](../07-玩法循环/工作.md) |
| **即时能力** | 移动一格、拾取等；不占工作进度 | 队伍系统、回合与行动表 |

**响应不是行为通道。** 响应仅负责**检测**；检测通过后，接续执行的仍是上表中的某一类行为。

### 作用（对行动调度的修正）

**作用**不是行为通道，也不改变世界资源或耐久；它修正**行动主体**在本回合内的**行动调度**（是否继续推进当前行动 / 工作），规则见 [工作中断与恢复](../07-玩法循环/回合与行动表.md#工作中断与恢复)。

| 作用（示例） | 说明 | 典型来源 |
|--------------|------|----------|
| **无** | 行为结算后，目标行动环节照常继续 | 环节结束后才触发的检测器；纯告警类行为 |
| **意外打断** | 目标本回合**不继续推进**当前行动；工作完成度**保留** | 响应检测器在**目标行动环节中**结算行为；遇敌；环境效果等 |
| **行动打断** | 目标行动 / 工作**暂停**（`suspended`）；完成度保留 | 交战占用行动、强制位移等 |

- **行为**改局面（伤害、进度、物品）；**作用**改调度（本回合还动不动）。
- 检测器配置须分别声明 **`trigger_behavior`**（走哪条行为通道）与 **`on_resolve`**（结算后施加何种作用，可为 `none`）。
- **意外打断**不限于响应：遇敌、环境效果等同理；响应只是最常见触发场景之一。

### 响应检测与行为触发

**响应**指附在环境、设施、单位等对象上的**条件检测器**：

| 环节 | 说明 |
|------|------|
| **检测器配置** | 监听时机、触发条件，以及 **`trigger_behavior`**（战斗 / 工作 / 即时能力及其参数）与 **`on_resolve`**（作用，如 `none` / `意外打断` / `行动打断`） |
| **监听窗口** | 通常在**其他行动主体**的 [行动阶段](../07-玩法循环/回合与行动表.md#回合阶段) 内，当其执行符合条件的行动时评估 |
| **触发** | 条件满足 → **`trigger_behavior`** 走行为通道结算 |
| **结算后作用** | 按 **`on_resolve`** 修正**目标行动主体**的行动调度（非行为通道内的一步） |

草稿将「响应」与「战斗、工作」并列列为行为三分，正式口径为：**战斗、工作（及即时能力）是行为通道；响应是触发这些行为的检测层。**

**示例（陷阱）**

| 配置项 | 内容 |
|--------|------|
| **检测器** | 陷阱（设施层）；监听：某单位在**自己的**行动环节中，从陷阱**相邻格**移动进入陷阱所在格 |
| **`trigger_behavior`** | **战斗**通道：对目标造成伤害；消耗该陷阱耐久 |
| **`on_resolve`** | **`意外打断`** @ 目标行动主体（本回合不继续推进移入陷阱的该步行动） |

**示例（哨塔，骨架）**

| 配置项 | 内容 |
|--------|------|
| **检测器** | 哨塔（设施层）；监听条件 **待定** |
| **`trigger_behavior`** | **待定**（如战斗通道攻击，或即时能力更新视野） |
| **`on_resolve`** | **待定**（告警-only 可为 `none`；交战占用行动可为 `行动打断`） |

同一行动环节中，多个检测器同时满足条件时的**检测顺序**，以及由此触发的多条行为之间的**结算顺序**，见待确认事项（sy-07）。

#### MVP 已落地边界（2026-06-27）

本期已落地的是**检测层骨架与分发接点**，不是完整检测器清单：

| 项 | 状态 | 说明 |
|----|------|------|
| `IResponseDetector` 接口 | 已实现 | 检测器只做条件判断并返回 `trigger_behavior` + `on_resolve`（对应 `ResolveEffect`） |
| `ResponseCheckPipeline` | 已实现 | 执行“先检测，后分发” |
| `BehaviorDispatchPort` | 已实现 | 将命中的行为触发发送到行为通道入口 |
| 意外打断接入 | 已实现 | `on_resolve=InterruptCurrentAction` 时进入中断流程，保留工作进度 |
| 检测器完整清单与全序规则 | 未实现 | 继续由 sy-07 跟踪 |
