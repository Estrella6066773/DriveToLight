## 目标

明确人口作为核心资源的承载方式、分配规则，以及**人口迁移**作为独立挑战玩法的定位。

## 范围

- **包含**：**居民与工作人口**、城区承载、玩家城区工作分配、人口分配原则、**人口归属（绑定领袖）**、迁移玩法定位、与队伍组建的关系。
- **不包含**：迁移具体数值、事件表；**城市管理系统**的 UI 线框与存储策略细则（见 [城市管理系统](./城市管理系统.md)、sy-23）。

## 详细说明

### 人口不是消耗品

- **人口**计入 [四种核心资源](./四种核心资源.md)，但**不因日常工作或编制分配而「消耗掉」**总量（战斗减员、事件损失、**周总结粮食不足导致的减员**等另计）。
- 城市人口池同时支撑：**居民**安置、**城区运作**劳动力、外出**队伍编制**、**随行人员**等用途；**人口总量**、**住宅承载**、**编制占用**分轨计数（见 [人口与住宅](#人口与住宅已定)）。
- **[食物](./四种核心资源.md)** **维持**活跃人口；**周总结**时未分到粮食 → **半数减员**（见 [粮食与周总结（草稿）](../../01-草稿/归档/粮食与周总结/README.md)）。

### 人口与住宅（已定）

**人口**与**住宅**是两套**独立**账，**不要**混为一谈：

| 维度 | 说明 | 主要绑定 |
|------|------|----------|
| **人口（总量）** | 城市领袖管辖池中的**人头**；战斗减员、粮食不足减员、事件损失等才减少总量 | **城市 / 领袖**管辖池 |
| **住宅（承载）** | 每座城区可安置的**居民上限**（基础承载 + 屋舍等）；谁「住在」哪座城区 | **城区**（见 [城区承载与居民安置](#城区承载与居民安置)） |
| **编制占用** | 外出 [队伍编制](#队伍与人口) 从人口池**占用名额**；**不**减少人口总量 | **队伍** + 来源 **城区**（住宅仍占用） |

- **创建队伍**：从城市人口池**占用编制**，**不是**把人口从城市或城区**移走**。
- 编组人员**仍归属**其绑定城区（`home_city_ref` / 住宅安置格），**仍占用**该城区**住宅容量**；即便整队在外执行任务，**不**视为**从住宅迁出**、**不**释放住宅槽位。
- **转移人口**（迁移玩法、跨城区**改换住宅安置** / **从住宅迁出**）与**编组外出****分轨**——后者**不**算从住宅迁出（见 [领袖与势力 · 势力城区管辖](../05-城市与领袖/领袖与势力.md#招募后势力城区管辖已定)）。

### 居民与工作人口

| 概念 | 说明 | 主要绑定 |
|------|------|----------|
| **居民** | 在城区**有住宅安置**的人口；占用该城区**居民承载**上限；城区 UI 显示人数以此为准（**含**当前在编组外出、仍占本城区住宅者） | **城区**（住宅安置） |
| **工作人口** | 承担 [城区运作](../03-图层与地点/建筑层/运作与居民.md#城区运作与居民人口)、城内 [工作](../07-玩法循环/工作.md) 的劳动力 | **玩家指定**：每城区由**哪一类人口**上岗（见下节） |
| **队伍编制** | 从城市人口池**占用**的编组名额；人员**不**离开来源城区之住宅归属 | **队伍** ← 来源 **城区** / 城市池（见 [队伍与人口](#队伍与人口)） |

- **城区上显示的人口 = 有住宅安置在该城区的人口**（含编组在外者），**不代表**当前在岗的**工作人口**。
- 城区要维持供能与运转，须有足够符合条件的**工作人口**到岗；仅有居民而无正确类型的运作人力时，城区可**停摆**或降级（细则 **待定**）。

### 废墟与居民迁出

状态为**废墟**的城区（含 **结构完整度低于 20%** 强制转入，见 [城市模块化 · 废墟](../03-图层与地点/建筑层/城区总览.md#废墟)）：

- **无法工作**：不参与城区运作，不消耗运作人力配置。
- **不可迁入**：玩家**不能**向该城区安置新居民。
- **仅可迁出**：已在该城区的居民**只能迁出**至其他**状态正常**且有空余承载的城区；迁出完成后该废墟城区居民数应为 0（**待定**：是否允许暂留无人废墟占格）。
- **招募 · 未效忠例外**：recruited 势力城废墟 **不适用**上条迁出特权——**禁止从住宅迁出**优先（见 [领袖与势力 · 废墟 × 招募](../05-城市与领袖/领袖与势力.md#废墟--招募--未效忠已定)）；须**修复**或维持现有人口安置。
### 玩家：城区工作分配

在**玩家移动城市**经营范围内，**工作区**（模块或设施）与**人口分配（住房）**由 [城市管理系统](./城市管理系统.md) 统一配置：

- 玩家须管理移动城市层面的**工作安排**（与 [工作](../07-玩法循环/工作.md)、[回合与行动表](../07-玩法循环/回合与行动表.md) 联动）。
- **特殊城区**：为**模块工作区**指定运作人口类型；**一般城区**：为各**设施工作区**指定（**同一套 UI**）。
- 不同城区可指定不同人口类型，也可在多个城区复用同一类型；是否允许同一类人口同时覆盖多城区、上限与冲突规则 **待定**（sy-23；占位表见 [城市管理系统 · 人力类型名单占位表](./城市管理系统.md#人力类型名单占位表占位--待策划定案--sy-23)）。
- **居民安置**（哪类居民住在哪个城区）与**工作人口指定**在管理系统内**分开配置**（见 [城市管理系统 · 三类分配](./城市管理系统.md#三类分配核心职能)）。

### 非玩家城市

- 外部城市每城只有 **1 名城市领袖**（见 [领袖与势力](../05-城市与领袖/领袖与势力.md)），**不向玩家暴露**城区级人口分配。
- 城区运作、居民与劳动力由领袖**统管抽象**；玩家与之交互时只感知**领袖关系**与结果，不操作其内部工作编制。

### 城区承载与居民安置

- **每个城区**（状态**正常**）均有 **基础居民承载**，由 `district_type_config.base_resident_cap`（字段名 **待定**）配置；**不**因未建任何设施而为 0（见 [城区总览 · 居民承载](../03-图层与地点/建筑层/城区总览.md#居民承载)）。
- 在 **一般城区** 内建造的 [屋舍](../03-图层与地点/设施层.md#屋舍) 设施可在基础上限之上 **叠加**额外承载。
- **居民承载上限** = 基础 + 屋舍等加成；获得**新城区**仍是扩充城市总容量的主要途径之一（来源 **待定**，见 sy-14）。
- 城区为**稀有资产**，不可通过常规建造获得（见 [城市模块化](../03-图层与地点/建筑层/README.md)）。

### 队伍与人口

- **创建队伍** / **补员** = 在**城区**上推进的多回合 [工作](../07-玩法循环/工作.md)（`form_team` / `reinforce_team`，对照 **2** 回合）；工作**完成后**方从城市人口池**占用编制**，见 [队伍系统 · 创队与补员](../06-单位与交战/队伍系统.md#创队与补员已定)。
- **创队至少 1 人**；单次编入人数 **≤** 模板编制人数（**默认人数 = 编制上限**；人数比分母同此值，**不是**必选满编数）。
- 编组人员**仍归属**其**住宅所在城区**（及 `home_city_ref` 所指向的城市）；**仍占用**该城区**住宅容量**，直至解散、减员或**改换住宅安置**（迁移玩法，见 [人口迁移](#人口迁移玩法定位)）。
- 另须满足 [队伍资产](../06-单位与交战/队伍系统.md#队伍资产与组建门槛) 组建门槛（清单 sy-21）。
- 人数比影响视野、负载、工作效率与战斗表现，**不**影响移速；见 [队伍系统 · 默认编制与人数影响](../06-单位与交战/队伍系统.md#默认编制与人数影响)。
- **粮食周总结**：编制在**队伍容器**吃**载荷粮食**；**不算**城区 L1 城市仓库切分，但**仍占**绑定城区**住宅**（见 [粮食与周总结 · §2.6](../../01-草稿/归档/粮食与周总结/粮食与周总结-已定案详述.md)）。

#### 编组 · 单一人口类型（已定）

每支队伍**仅**含**一类**人口，**禁止**混编，以免 [人口归属](#人口归属) Buff 混杂：

| 项 | 口径 |
|----|------|
| **原则** | 编组时锁定 **`population_type_id`**（程序字段 **待定**）；全队归属 Buff、商队能力判定均读此类型 |
| **创队** | 玩家在城区发起 `form_team`；指定模板、**单一** `population_type_id` 与 **1～max** 人数；UI **禁止**跨类型混选；工作完成后写入 map |
| **补员** | 在编队伍在**当前格**发起 `reinforce_team`；从**相邻格**来源池划拨 **≤ 剩余编制上限** 的同类型人数 |
| **与模板** | [队伍模板](../06-单位与交战/队伍系统.md#队伍模板概览默认预设) 决定**能力**（侦察 / 运输 / 武装等）；人口类型决定**归属 Buff**——二者独立，但编制须来自**同一**类型 |
| **系统编组** | 商队等：贸易比较等规则先 [选定类型](../05-城市与领袖/领袖与势力.md#商队履约已定)，再按 [城区来源人数](#编组--城区来源人数已定) 写入 map |
| **枯竭** | 创队后 `population_type_id` **不变**；管辖池内该类型**无**未预占用可划拨编制时 **不可**再发起创队 / 补员（**不**自动改类型） |

#### 编组 · 人口预占用（已定）

凡 [创队](#编组--单一人口类型已定) / [补员](#补员--相邻格人员来源已定) 等**指令**为某次编组**指定**从来源池划拨的人数，在指令**受理当下**即从对应池 **预占用**（`reserved_headcount`）。

**先下指令者占用，没有优先级**：商队（系统）先下达并受理 `form_team` → 商队预占用；玩家先下达并受理创队 / 补员 → 玩家预占用。二者**不**设谁优先，仅看**哪条指令先完成受理**。

| 项 | 规则 |
|----|------|
| **预占用时机** | 指令**受理**瞬间（玩家确认创队 / 补员，或系统生成商队 `form_team` 并写入划拨方案） |
| **互斥** | **已预占用**人口 **不可**被其它行动使用（含其它创队 / 补员、迁移、归属转化、随行接纳等，除非该行动显式释放同一预占用票据） |
| **争用同一池** | **无**商队 vs 玩家优先权；**先受理**的指令占用，**后受理**者若可用量不足则 **不可受理** |
| **可用量** | 各来源池 **剩余可编 = 池总量 − 已编 − 已预占用** |
| **工作完成** | 预占用**落实**为 `district_headcount_map` / 编制占用（及住宅槽位）；预占用票据销账 |
| **中断 / 取消** | 工作未落实前取消或 `on_interrupt_disposition` 清除条目时，**释放**对应预占用；**不**写入编制 |
| **UI** | 各来源池展示 **可用 = 可编 − 已预占用**；不足则指令 **不可受理** |

程序占位：`PopulationReservationState` / 指令条目 `reserved_allocation_json`（见 [队伍与战斗数据结构](../../03-程序设计/03-数据字典/队伍与战斗数据结构.md)）。

#### 编组 · 城区来源人数（已定）

**创建队伍**与**补员**时，须**按城区**记录本队编制中各有多少人数来自哪一座**住宅所在城区**。**不**追踪具体哪一名人口，仅维护各城区**人数**聚合：

| 项 | 口径 |
|----|------|
| **记录内容** | 每支队伍维护 **`district_headcount_map`**：`{ district_id → 人数 }`；与 `current_headcount` 之和一致 |
| **追踪粒度** | **不**记录个体；仅统计各城区贡献的编制**人数** |
| **编组时** | 指令受理时 **预占用**（见 [编组 · 人口预占用](#编组--人口预占用已定)）；工作**完成**后写入各城区**来源人数** |
| **减员时** | 每损失 **1** 人：从 `district_headcount_map` 中**按各城区当前人数权重随机**抽取 **1** 个城区，该城区条目 **−1**；同步更新队伍人数、该城区**居民 / 住宅占用** |
| **随机** | 权重 = 该城区在 map 中剩余人数 / 全队剩余人数；**0** 人城区不参与抽取 |
| **确定性** | 程序用对局 **`game_seed`** + 事件序（战斗实例 ID、周总结批次等）驱动可复现随机（字段 **待定**） |
| **补员** | 新增编制**追加**写入对应 `district_id` 计数；须与队伍 `population_type_id` **一致**（见下表） |
| **解散** | 按 map **原样释放**各城区编制占用与住宅槽位（**不**再随机） |

**补员 · 相邻格人员来源（已定）**

**补员**（`reinforce_team`）的人员**只能**从队伍**当前格**与**地图相邻格**（六邻格，与单位移动邻接一致）内的下列**来源池**划拨；各池仅计 **未预占用**可用量（见 [编组 · 人口预占用](#编组--人口预占用已定)）。玩家在编组 UI 中指定各池人数；总和 **≤** 本次补员上限且 **≤** `max_headcount − current_headcount`。

| 来源池 | 所在格 | 说明 |
|--------|--------|------|
| **城区编制** | 相邻格上属于玩家（或 `home_city_ref` 绑定城市）的**城区占格** | 从该城区可划拨编制计入 `district_headcount_map`；须该城区有住宅承载与可编人口 |
| **随行人员** | 相邻格上**任意玩家队伍**的 `EscortAccompanyingState` | 含**本队**随行人员；转入编制后自载荷随行池扣减 |
| **征兵办闲置** | 相邻格上的 **[征兵办](../荒野地点/征兵办.md)** 设施 | 已自村镇储量提取、**尚未**接纳入城或转编制的 **`idle_headcount`**（见 [recruitment-office-data](../../03-程序设计/03-数据字典/recruitment-office-data.md)） |

**人口归属限制（已定）**

| 项 | 规则 |
|----|------|
| **类型** | 各来源池人口的 `population_type_id` **须与**目标队伍锁定类型**完全一致**（禁止混编） |
| **归属** | 须为 `home_city_ref` 绑定城市之**任职领袖**管辖下**可编入**的归属；**敌对 / 未转化**归属 **不可**直接划入 |
| **无归属** | [征兵办](../荒野地点/征兵办.md) 闲置或储量人口为 **`pop_unaffiliated`** 时，**不可**直接编入已锁定具体归属类型的队伍；须先 [人口归属转化](../05-城市与领袖/领袖与势力.md#人口归属转化独立功能--已定框架) 或接纳入**相邻城区**后再从城区池补员 |
| **写入 map** | 自**城区**补员 → 写入对应 `district_id`；自**随行人员** → 写入该随行人员绑定的来源 `district_id`（无绑定 → **保留在携带队伍中作为随行人员**）；自**征兵办闲置** → 写入征兵办绑定村镇对应的接收城区，住房不足时人口保留在携带队伍中作为随行人员 |

| 场景 | 规则 |
|------|------|
| **玩家手动补员** | 在补员 UI 指定**相邻格**内各来源池及人数；须通过归属、类型校验且池内 **未预占用**量足够 |
| **系统创队**（商队等） | 城区 `form_team`；按 [商队履约](../05-城市与领袖/领袖与势力.md#商队履约已定) 选定城区与人数；**先受理者占用**，与玩家创队**无**优先权 |
| **系统补员**（无 UI） | 在相邻格来源池中，自 **`population_type_id` 未预占用可用量最大**的池追加；不足则 **不可**补员 |

**创队 · 城区来源（已定）**（与补员分轨）

| 场景 | 规则 |
|------|------|
| **玩家手动创队** | 在**城区**发起 `form_team`；从该城区及规则允许的绑定城市内城区划拨；UI **禁止**跨类型混选；受理时 **预占用** |
| **系统创队**（商队等） | 城区 `form_team`；自 [商队履约 · 组队城区](../05-城市与领袖/领袖与势力.md#商队履约已定) 选定城区与人数；**先受理者占用** |

- **队伍减员**（交战、粮食未分到、事件等）均走同一 **随机落点** 规则；落点城区计入 [势力系统 · 单位轨 / 城区轨](../05-城市与领袖/势力系统.md#分轨累计已定) 中**城区居民减员**一侧（因住宅在该城区）。
- **`home_city_ref`** 仍为**城市级**关系事件路由；**城区来源 map** 供住宅、承载、关系统计与 UI 展示，**不**替代 `home_city_ref`。
- 程序字段见 [队伍与战斗数据结构 · `district_headcount_json` / `population_type_id`](../03-程序设计/03-数据字典/队伍与战斗数据结构.md#表-9team_instance_state队伍实例状态)。

### 随行人员

> **废止**「载荷迁徙人口」口径；由 **随行人员** 替代。详见 [粮食与周总结 · §2.10](../../01-草稿/归档/粮食与周总结/粮食与周总结-已定案详述.md)。

| 项 | 规则 |
|----|------|
| **定义** | 随队伍移动、**尚未入城区驻留** 的玩家人口；**不是**队伍编制 |
| **工作** | **不参与**队伍工作；**不计入** `GetWorkableTeamHeadcount` |
| **载荷** | 占用队伍载荷容量；提供默认人口物品载荷 |
| **粮食** | 周总结计入队伍容器需求（与编制分列，同吃载荷粮食） |
| **归属** | 队伍**接纳时**人口**已属玩家** |
| **住宅** | 自**来源获取人口时**须**预先规划安置**（sy-14） |

人口可从 [村镇储量](./荒野地点.md)（**无归属**人口；经 **[征兵办](./荒野地点/征兵办.md)** 提取，见 [村镇](./荒野地点/村镇.md)）进入城区驻留，或经队伍接纳为随行人员。

### 人口归属

- **人口归属**（文化 / 势力身份）与人口**总量**分开管理：归属决定加成、关系与部分事件，不因日常编制分配而消失。
- 归属**绑定在城市领袖**名下，由领袖管辖的城市人口池继承；**不**按城区直接绑定。归属转化是领袖 / 人口**独立功能**（**不是**城区能力）；学院仅放宽可转化目标——见 [领袖与势力 · 人口归属转化](../05-城市与领袖/领袖与势力.md#人口归属转化独立功能--已定框架)。
- 队伍编制从某城市人口池划拨时，继承该城当前**任职领袖**下的归属规则；编组须遵守 [编组 · 单一人口类型](#编组--单一人口类型已定)（**禁止**混编）。

### 人口迁移（玩法定位）

**人口迁移**指人口在不同定居点、城区或城市之间的流动，预期成为颇具挑战的**子系统玩法**，包括但不限于：

- 从 [村镇](./荒野地点/村镇.md)（**无归属**储量，经 **征兵办** 提取）、外部城市招募或接收流民；
- 停泊时与外部据点的人口交换；
- 章节事件驱动的大规模迁徙；
- 与 [势力系统](../05-城市与领袖/势力系统.md) 关系挂钩的接纳 / 驱逐；
- 流入人口纳入目标城市**城市领袖**管辖池，归属按 [领袖与势力](../05-城市与领袖/领袖与势力.md) 规则确定或转化。

当前仅确立**需要独立设计**；具体规则、触发条件与代价见 sy-14。

### 与停泊 / 航行的关系

- **停泊**：移动城市座落世界地图，可与外部据点发生**物理接触**层面的人口与物资交互（见 [地图与移动 · 停泊与航行](../02-地图与世界/地图与移动.md#停泊与航行)）。
- **航行**：城市**不占**世界地图格；人口与物资的**面对面**交换不可用；远程交互与在途队伍规则见 **sy-19**（[地图与移动 · 航行态](../02-地图与世界/地图与移动.md#航行态占格单位边界与禁用功能-sy-19)）。
