## 目标

明确首版**发行平台**、**画面维度**与**主要操作方式**，为渲染管线、输入模块与 UI 布局提供对照。

## 范围

- **包含**：平台范围、3D/2D、相机观察方式、鼠标点击操作原则。
- **不包含**：具体键位表、UI 线框、渲染技术选型（URP/HDRP 等，见程序设计后续文档）。

## 已定

| 主题 | 结论 |
|------|------|
| **发行平台** | **PC**（Windows）；首版不面向主机、手游 |
| **画面维度** | **3D 场景**；玩法逻辑仍基于正六边形网格与图层栈，不是 2D 贴图平面 |
| **主要操作** | **以鼠标点击为主**；编辑指令表、行动表、选中单位/城区/格子、确认菜单等均须可用纯点击完成 |
| **镜头类型** | **策略向俯视角**（自上方观察地图与城市），参考《文明》类回合策略的可读性，而非动作游戏的跟拍镜头 |
| **默认相机预设** | 以**玩家移动城市（主城）**为观察中心；镜头**朝向地面**（俯视主城占格）；视线与**水平面夹角 35°**（非正顶视） |
| **俯仰可调范围** | **0°～90°**（相对水平面；0° 为平视高度、90° 为正顶视）；默认可复位至 **35°** |
| **相机拖拽倍率** | 中键拖拽旋转采用倍率映射，当前默认 **1.5**；旋转与俯仰均围绕地图中心坐标（pivot） |
| **相机平移速率上限** | 拖动地图平移时设速率上限，**默认镜头距离下为每秒 10 格**；上限随**屏幕缩小倍率**等比放大（镜头拉远一倍，上限同倍放大；以默认镜头距离为 1× 参照），避免快速拖拽时产生眩晕或跳帧 |
| **相机插值平滑** | 平移（pivot）与缩放（距离）均采用**插值平滑移动**：输入先写入目标值，相机按阻尼方式跟随目标值，跟随速度同样受平移速率上限约束；平滑时长参数的默认值**待调校**（sf-33） |
| **相机平移语义** | 玩家"移动摄像头"定义为**移动中心坐标（pivot）**，非移动相机本体；旋转与缩放继续围绕该中心 |
| **场景与镜头** | **仅一套世界地图观察**；不存在单独的「城市内部视图」——移动城市即由 [城区](../03-图层与地点/建筑层/城区总览.md) 组成，在底图上占格（见 [地图与移动](../02-地图与世界/地图与移动.md)） |
| **指挥交互（MVP）** | 左键点格后，围绕该格显示悬浮指挥 UI：左侧为“此格可选目标列表”，右侧为“目标可用命令列表”；若命令存在多个目标，再在右侧展开目标列表。列表默认隐藏，仅在选中有效格点后激活；UI 结构来自预制体，不在运行时临时造节点。选中单位后需先点“下达命令”进入指挥模式，随后可继续点选其他格/目标并在右侧显示可交互项；空格也支持“移动到此格”。指令支持“添加到指令表 / 重写指令表”。选中单位时，其指令目标、预判路径与“本回合可抵达范围（红点）”在棋盘上可视化，并在调试面板展示逐条执行计划。左右栏中下方增加状态栏，实时显示当前目标正在执行的事项（例如城市正在开拔/安扎）；若目标处于工作中，则额外展示工作进度与待进行回合数。状态栏的回合统计按**当前指令**单独计算，显示“当前指令预计 N 回合完成”。右键任意点按层级回退：先收起目标列表，再退出指挥，根层再右键则折叠 UI |

### 与玩法的关系

- [回合与行动表](../07-玩法循环/回合与行动表.md) 的「玩家指挥 → 玩家行动」流程以**点选 + 面板确认**为主，不依赖实时键鼠连招。
- [城市管理系统](../04-资源与人口/城市管理系统.md) 以**点选地图上的城区 → 打开管理面板**为入口，与上述操作方式一致。
- [地图与移动](../02-地图与世界/地图与移动.md) 的六边形占格、多层图层信息须在俯视下可辨；3D 用于表现城区高度、环境效果与 Sun-punk 氛围，**不改变**格点逻辑。

## 需求驱动指挥（工作计划）

除「选中单位再下达命令」的**单位先行**入口外，新增一套**需求先行**入口：玩家先在地图上对目标下达**需求**，系统据此创建一份**工作计划**，再由玩家指派完成方（也可不指派）。两套入口**并存**，底层都汇入 [队伍系统](../06-单位与交战/队伍系统.md) 的同一套指令表与 [工作](../07-玩法循环/工作.md) 机制，**不**另设第二套执行系统。

### 已定框架

| 主题 | 结论 |
|------|------|
| **入口关系** | 单位先行与需求先行**并存**；需求先行是新增便捷层，底层转为队伍指令表条目 |
| **工作计划对象** | 需求下达后生成一份工作计划，可**仅创建而不指派**（此时将信息交给 [游戏日志](#游戏日志)），或当场指派；未指派工作计划进入全局**工作计划清单**，**可跨回合保留** |
| **指派方式** | **手动为主**：二级菜单列出可承接的候选（合格的现有队伍 / 新建队伍）供玩家选取；另提供可选的**自动指派**（派最近的合格队伍） |
| **新建队伍** | 选择新建队伍时打开**队伍创建页面**，须配置：**队伍类型**（从符合条件、能执行该任务的单位中筛选）、**人口类型**（人口归属）、**人数**；创建后自动组建队伍并派发指令 |
| **指派修改** | 已指派的工作计划可**修改指派对象**：从原单位指令表中**仅移除本条计划对应的指令**（不清空整张指令表），再为新单位按第三层规则添加或替换 |
| **行动表占位** | 每条工作计划（含对应指令）在 [行动表](../07-玩法循环/回合与行动表.md#行动表) 中**占一个独立位置**；被指派的单位**自动进入该占位**，不再另占一行 |
| **第三层加入方式** | 指定某支具体队伍后弹出**添加 / 替换**选项，顺序**添加在上、替换在下**；无可替换时直接添加跳过第三层（如新建队伍） |
| **首版需求类型** | **建造设施**、**运维设施**、**运输 / 寻回物资**；其余类型待定（sy-37） |
| **底层复用** | 指派后落入所选队伍指令表，按 [工作效率与工作量](../07-玩法循环/回合与行动表.md#工作效率与工作量) 逐回合推进；工作对象 / 被工作对象、可暂停、完成度、数值通道均沿用现有 [工作系统](../07-玩法循环/回合与行动表.md#工作中状态) |

### 流程

1. **下达需求**：玩家选取目标（格 / 区域 / 资源点 / 设施）→ 发起一类需求。
2. **生成工作计划**：系统创建工作计划（未指派状态），记录需求类型、目标与参数，纳入工作计划清单。**至此可结束**——玩家稍后再指派。
3. **指派完成方**（可选，可在创建后随时进行）：二级菜单列出候选队伍（含新建队伍）；玩家手动选取，或选择自动指派。
   - **选择新建队伍**：进入队伍创建页面，配置类型 / 人口 / 人数后自动组建并派发指令。
4. **转为指令执行**：指派后工作计划落入所选队伍指令表（按第三层规则添加或替换）。

### 仅创建、不指派

玩家可**只创建工作计划而不当场指派**：

- 未指派的工作计划保留在全局工作计划清单中，可跨回合暂存。
- 此时向 [游戏日志](#游戏日志) 发出一条**待办**消息，提醒存在待指派的未指派需求。
- 稍后可从工作计划清单中重新进入指派流程。

### 指令 vs 需求：UI 区分（已定）

玩家在详情面板（如设施详情、工作区面板）选定工作时，系统按条件给出不同的操作入口：

| 条件 | 入口按钮 | 行为 |
|------|----------|------|
| **条件已满足**（所需资源已在城市仓库、人口充足等） | **下达指令** | 直接开始工作 |
| **条件未满足**（资源不在仓库、需外部运送等） | **下达需求** | 系统自动挂起，生成工作计划与运输筹备链，条件满足后自动衔接执行 |

两按钮须有**明显视觉差异**（颜色、图标、文案不同），以提示玩家当前是即时指令还是自动筹备链。

> 详见 [城市内资源迁移与城外运送](../07-玩法循环/工作.md#城市内资源迁移与城外运送已定)。

### 指派修改

已指派的工作计划**可修改指派对象**（如从队伍 A 换成队伍 B）：

- 从原单位的指令表中**仅移除本条计划对应的指令**，**不清空**该单位的其余指令。
- 此后按正常指派流程为新单位添加（或替换）该条指令。
- 修改指派时第三层选项（添加 / 替换）照常弹出，规则不变。

### 新建队伍

在指派阶段选择「新建队伍」时，进入**队伍创建页面**，须配置以下三项；配置完成后自动组建队伍并派发指令：

| 配置项 | 说明 |
|--------|------|
| **队伍类型** | 从**符合条件**的模板中筛选——仅列出**具备执行该任务所需能力**的单位类型 |
| **人口类型** | 选择队伍编制的人口归属（`population_type_id`），遵守 [编组 · 单一人口类型](../04-资源与人口/人口与迁移.md#编组--单一人口类型已定) |
| **人数** | 指定编制人数，**至少 1 人**，**不超过**模板编制上限 |

- 新建队伍本身走 `form_team` 工作管线（见 [创队与补员](../06-单位与交战/队伍系统.md#创队与补员已定)），队伍就绪后工作计划作为第一条待办（指令表为空，直接添加跳过第三层）。

### 行动表占位

工作计划与 [行动表](../07-玩法循环/回合与行动表.md#行动表) 深度集成：

| 规则 | 说明 |
|------|------|
| **占位** | 每条工作计划（含对应指令）在玩家行动表中**独立占一个位置**（`order_index`），与直接指挥的单位行动表条目同属一张表 |
| **单位入位** | 被指派的单位**自动进入该占位**，不再另占一行；一个单位被多条工作计划指派时，该单位在各条计划的占位处各自出现 |
| **未指派占位** | 尚未指派的工作计划也在行动表中占位（占位符形态），指派后该占位由对应单位填充 |
| **玩家调控** | 玩家在指挥阶段可像常规行动表条目一样**调节工作计划占位的顺序**（见 [调节顺序（插队）](../07-玩法循环/回合与行动表.md#调节顺序插队)） |

### 与现有入口的关系

- [队伍系统 · 快速建造（资源点）](../06-单位与交战/队伍系统.md#快速建造资源点) 是本体系的**已有实例**：均为「选目标 → 选完成方式 → 指派队伍（含新建）」，纳入统一的需求先行管线。
- 需求先行**不绕过**各需求类型的前置条件与工作时长（如建造条件），仅缩短「找队伍 → 下指令」的操作路径。

### 资源预留与预占用

| 阶段 | 行为 |
|------|------|
| **仅创建工作计划**（未指派） | **不预留任何资源**。工作计划只是待办备注，不阻断其他指令使用同一批人口/物资。 |
| **指派到队伍时** | 才触发预留——与现有 [编组 · 人口预占用](../04-资源与人口/人口与迁移.md#编组--人口预占用已定) 管线一致（先下指令者先占）。 |
| **新建队伍** | `form_team` 工作下达时触发预占用，属于 `form_team` 管线自身规则，无需工作计划额外处理。 |

### 自动指派判定

| 判定 | 规则 |
|------|------|
| **合格条件** | 队伍须具备该任务类型对应的 `Capability` tag（建造运维 = 建造能力、运输 = 搬运能力）；`is_alive=true`；**不在**不可恢复的占用状态（如交战中） |
| **距离** | 按六边形**直线距离**排序（不计算寻路），取最近 |
| **同距离** | **无待办队伍优先**（`command_queue` 为空或仅有当前执行中条目），其次按创建时间早的优先 |
| **结果展示** | 按距离排序的列表，各队伍旁显示距离格数与当前待办数，玩家仍可手动覆盖自动结果 |

### 取消工作计划

| 取消时状态 | 处置 |
|------------|------|
| **未指派** | 直接删除工作计划 + 清理行动表占位；若已向日志发送过提醒，同步清除对应日志条目。零惩罚。 |
| **已指派，队伍尚未开始本条** | 从队伍指令表**移除本条**（单条移除，不清空其余指令）；清理行动表占位。零惩罚。 |
| **已指派，队伍正在执行本条** | 按现有 [工作中断与恢复](../07-玩法循环/回合与行动表.md#工作中断与恢复) 规则处理——可暂停的工作保留进度、不可暂停的归零；处置为 `ended` 时本条结束；清理行动表占位。**不额外加惩罚**。 |
| **已指派，队伍已阵亡** | 见下文待补（G项）。 |

### 首版需求类型与二级参数

| 需求类型 | 主目标（一级） | 二级参数 | 默认值 |
|----------|---------------|----------|--------|
| **建造设施** | 所选格（资源点 / 可行格） | **设施类型** | 从该格合法建造清单选一个（同现有快速建造二级菜单） |
| **运维设施** | 所选格上的设施 | **运维投入**（可选） | 默认 = 标准运维工作量（`base_duration_turn`）；MVP 不开放超额运维 |
| **运输 / 寻回** | 目标格（有可拾取物） | **目的地仓库** | 默认 = 队伍归属城区的仓库节点；玩家可改为其它仓库 |

### 跨回合保留与超时提醒

- 工作计划**不过期**，由玩家自主管理。
- 工作计划创建后每过 **7 回合**未指派，触发一条**警告**到 [游戏日志](#游戏日志)，提醒玩家尚有未处理的需求；除提醒外无其它影响。
- 服务器不自动删除或回收工作计划。

### 被指派队伍阵亡

被指派的工作计划对应**指令所属队伍阵亡**时：

| 处置 | 说明 |
|------|------|
| **工作计划回退** | 工作计划回到**未指派**状态，重新纳入工作计划清单 |
| **行动表清理** | 原先在行动表中的占位标记为**无效**，待玩家重新指派后恢复 |
| **玩家提醒** | 向 [游戏日志](#游戏日志) 发出一条**报错**消息，标明工作计划名称 + 原指派队伍 + 回退状态 |

### 待补细则（剩余）

工作计划清单的界面、工作计划创建后指派前的**参数编辑**等**尚未定**，登记 [待细化追踪-系统 · sy-37](../../00-规范/待细化追踪-系统.md#当前开放项)；运行时对象与指派服务见 [待细化追踪-程序 · sf-35](../../00-规范/待细化追踪-程序.md#当前开放项)。

## 游戏日志

游戏日志是供游玩者了解游戏状态的**类控制台报告面板**，记录单位行动、系统提醒与紧急情况。分为三种消息等级和两种解决模式。所有**警告**和**报错**消息在日志列表中以**黄、红标记**标示，供玩家快速扫描异常。

### 消息等级与标记

| 等级 | 标识 | 列表标记 | 触发场景（示例） | 严重程度 |
|------|------|----------|------------------|----------|
| **日志** | `info` | **无标记** | 单位完成移动路径点、完成一项工作、使用一次能力 | 常规 |
| **警告** | `warn` | **黄色标记** | 工作计划 7 回合未指派、指定路径不可达、安排的任务无人前往完成 | 会产生影响但不紧急 |
| **报错** | `error` | **红色标记** | 单位遭袭、单位死亡、被指派队伍阵亡导致工作计划回退 | 紧急 |

### 解决模式

| 模式 | 说明 | 示例 |
|------|------|------|
| **已知** | 玩家**点击即视为已处理**，消息归档 | 单位到达路径点、建造完成 |
| **待办** | 点击**不能**直接解决；玩家须对底层问题做出处理，使其不再符合当前级别条件，消息才自动消失 | 工作计划无指派单位（须指派 / 取消后方消失）、路径不可达（须修改路径或取消指令） |

### 与游戏内提醒的关系

- 工作计划的提醒**统一走游戏日志**（待办消息）；**不另设地图图标**或地图级提醒。工作计划的指派、修改、取消等操作通过日志记录，玩家从日志列表中追踪和跳转。
- 工作计划**7 回合超时提醒**（见 [跨回合保留与超时提醒](#跨回合保留与超时提醒)）发出一条**警告**级**已知**消息（仅告知，不须玩家在日志中做额外操作）。

### 待补细则

游戏日志的界面布局、存储上限与归档策略、消息详情面板内容、**待办**消息的自动消失判定频率（实时 / 每回合末）等**尚未定**，登记 [待细化追踪-系统 · sy-38](../../00-规范/待细化追踪-系统.md#当前开放项)；运行时对象与日志服务见 [待细化追踪-程序 · sf-36](../../00-规范/待细化追踪-程序.md#当前开放项)。

## 相机（观察，非玩法）

**可以旋转地图并改变观察角度，但建议「有限调整」，不做完全自由的轨道相机。**

### 默认预设档（OPEN-036 已定）

| 参数 | 定案 |
|------|------|
| **观察中心** | **玩家移动城市（主城）**占格几何中心；城市移动或占格变化时，默认预设应能重新对准主城（**平滑过渡方式待定**） |
| **视线目标** | 镜头**朝向地面**（看向观察中心所在地平面） |
| **水平朝向（Yaw）** | 默认使**屏幕上方**对齐地图**纵向向上**（追日方向，见 [地图地理关系](../../04-设定/03-地点与场景/README.md#地图地理关系)） |
| **俯仰（Pitch）** | 视线与**水平面**夹角 **35°**（默认）；玩家可在 **0°～90°** 范围内调整 |

程序侧可记为：`pitch_elevation_deg` 默认 **35**，合法区间 **`[0, 90]`**（相对水平面）；默认 `look_at` 为主城中心地平面，默认 Yaw 使屏幕上方对齐卷轴向上。

**俯仰边界说明**

| 角度 | 含义 |
|------|------|
| **0°** | 与水平面平行（平视高度，不向下俯视） |
| **35°** | 默认预设（策略向浅俯视） |
| **90°** | 正顶视（视线垂直向下） |

### 场景口径（OPEN-038 已定）

- **不存在**单独的「城市内部视图」或分场景城市场景。
- **移动城市** = 多个 **城区** 实例 + 连接关系；停泊时每个城区对应底图（世界地图）建筑层上的占格，与 [城区总览 · 城区：定义与资产定位](../03-图层与地点/建筑层/城区总览.md#城区定义与资产定位) 一致。
- 相机始终观察**同一套底图**；选中、指挥、城区面板等均在该观察空间内完成，**不**切换至另一套镜头规则。

### 其他观察能力

| 能力 | 建议 | 理由 |
|------|------|------|
| **水平旋转（Yaw）** | **支持**；中键拖拽、UI 按钮或 Q/E 等效操作均可；**复位**可回到上述默认预设（屏幕上方 = 卷轴向上 + 俯仰 35°） | 便于从不同方向查看占格形状与前方地图 |
| **俯仰（Pitch）** | 默认 **35°**；可在 **0°～90°** 内连续或分档调整；**复位**回到 35° | 边界已定；具体分档 **待定** |
| **缩放** | **支持**滚轮或 UI 按钮 | 兼顾全局路线与局部占格细节 |
| **格网朝向** | 可选：旋转时吸附至 60° 倍数，或保持 UI 罗盘指示「卷轴向上 = 追日方向」（sy-17） | 避免玩家失去方位感 |

**原则**：相机只改变**观察**，不改变格子坐标、移动规则或胜负判定。

## 输入分层（程序侧对照）

| 层级 | 说明 |
|------|------|
| **必选（点击）** | 选中、确认、取消、打开面板、编辑指令/行动表、下达工作；左键点格进入格周悬浮指挥（目标/命令/命令目标）。当指针命中列表 UI 时，不再执行 3D 射线选中 |
| **可选（便捷）** | 键盘快捷键、滚轮缩放、拖拽平移/旋转相机；首版可不绑定快捷键，但架构须预留 |
| **不要求** | 手柄、触屏手势、双摇杆移动 |

`Yanxu.Input` 模块以**点击射线命中格点 / 单位 / UI** 为一等公民；相机操作走独立通道，不与指令下达共用同一套「选中即执行」逻辑。
