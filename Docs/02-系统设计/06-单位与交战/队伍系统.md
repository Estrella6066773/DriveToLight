## 目标

定义玩家可组建的远征队伍模板、能力组合与执行规则，支撑探索、运输、资源开发与战斗玩法。

## 范围

- **包含**：队伍类型、职责、**单一人口类型编组**、**队伍资产与组建门槛**、**人数对比率与能力衰减**、基础限制、战斗身份判定、队伍 AI 执行规则。
- **不包含**：具体数值平衡、逆风保护细则、程序实现（回合阶段见 [回合与行动表](../07-玩法循环/回合与行动表.md)）。

## 详细说明

### 概述

队伍系统是玩家将城市影响力扩展到荒野的核心机制。玩家组建各类队伍外出执行探索、运输、建设等任务。

> **详细单位定义**（属性、视野能力等）见 [单位类型与视野](./单位类型与视野.md)。
> 
> **视野同步与侦察机制**见 [视野系统](./视野系统.md)（**不存在侦察时间差**；侦察塔=野外战略设施）。

### 能力化设计原则

- 程序不写死“某队伍只能做某件事”，只提供可启用能力的执行通道。
- 队伍行为由 ScriptableObject 配置决定；开发者在引擎内手动配置能力与参数（可由 MCP 辅助执行）。
- 队伍“种类”在实现上视为预设模板，本质是能力组合。

**配置入口（系统策划）**：四类默认能力的自然语言说明见 [能力说明.csv](../../03-程序设计/03-数据字典/tables/shared/能力说明.csv)；填写规则见 [策划可读配置表规范](../../00-规范/策划可读配置表规范.md)。程序字段与 SO 结构见 [队伍与战斗数据结构](../../03-程序设计/03-数据字典/队伍与战斗数据结构.md)。

| 项 | 规则 |
|----|------|
| **原则** | 每支队伍**仅**含**一类**人口；**禁止**混编 |
| **Buff** | 人口归属 Buff 只读创队时锁定的 `population_type_id`；补员**不**改变类型或 Buff |

### 创队与补员（已定）

玩家在**城区**发起**创建队伍**或**补员**时，**不**即时划拨编制，而是下达一项多回合 [工作](../07-玩法循环/工作.md)：**队伍在该城区完成对自身编制的组建**。

| 项 | 规则 |
|----|------|
| **工作类型** | **创建队伍** `form_team` · **补员** `reinforce_team` |
| **被工作对象** | **创队**：目标 **城区实例**（`work_subject_ref` = `district_id`）。**补员**：**队伍实例**（`work_subject_ref` = `team_id`） |
| **工作对象** | **队伍实例**——创队时为**待组建**队伍；补员时为**在编**队伍（在**当前格**完成补员工作） |
| **对照时长** | 各 **2** 回合所需工作量（`base_duration_turn=2`；实际回合受 [工作效率与工作量](../07-玩法循环/回合与行动表.md#工作效率与工作量) 影响） |
| **创队人数** | **至少 1 人**即可发起；玩家指定本次编入人数 **≤** 模板编制人数（**默认人数 = 编制上限**，见下；**不**要求一次满编） |
| **创队人员来源** | 在**城区**上发起；从该城区（及创队规则允许的绑定城市内城区）划拨编制，见 [人口与迁移 · 编组](../04-资源与人口/人口与迁移.md#编组--城区来源人数已定) |
| **补员人数** | 追加人数 **≤** 编制人数 − `current_headcount`；须与创队同 `population_type_id` |
| **补员人员来源** | **仅**从队伍**当前格及地图相邻格**（六邻格，与移动邻接一致）内的下列池划拨，见 [补员 · 相邻格人员来源](../04-资源与人口/人口与迁移.md#补员--相邻格人员来源已定) |
| **补员工作格** | 在队伍**当前所在格**执行 `reinforce_team`（**不要求**回城；被工作对象 = **队伍实例**） |
| **完成时** | 预占用**落实**：写入 / 更新 `district_headcount_map`、占用编制与住宅；队伍由**待组建**转为可出征（创队）或 `current_headcount` 增加（补员） |
| **中断** | **可暂停**；取消或清除未完成工作时 **释放**预占用；**不**落实编制 |
| **预占用** | **先下指令者占用**：商队与玩家创队 / 补员**无**优先权，以指令**受理**先后为准，见 [编组 · 人口预占用](../04-资源与人口/人口与迁移.md#编组--人口预占用已定) |
| **门槛** | 仍须满足 [队伍资产](#队伍资产与组建门槛)、[单一人口类型](../04-资源与人口/人口与迁移.md#编组--单一人口类型已定) 与各来源池 **未预占用**可用量 |
| **商队** | 贸易触发的系统 `form_team` 与玩家创队共用预占用与 `form_team` 管线 |

- **工作计划 · 新建队伍**：当队伍通过 [需求驱动指挥（工作计划）](../01-核心体验/平台与操作.md#需求驱动指挥工作计划) 的「新建队伍」入口创建时，队伍创建页面额外提供**队伍类型筛选**（仅列出具备任务所需能力的模板）、**人口类型**与**人数**配置；创建后自动派发工作计划对应指令。

- **编制人数（已定）**：**默认人数 = 编制上限**（同一数值）。程序字段可仍写 `default_headcount` / `max_headcount`，但首版配置**必须相等**；人数比分母与创队 / 补员硬上限均取此值。
- **创队 / 补员**：可少于编制人数（至少 1 人），**不得超过**编制人数。

### 队伍资产与组建门槛

除 [人口编制](../04-资源与人口/人口与迁移.md#队伍与人口) 外，组建队伍实例还须满足 **队伍资产** 条件（框架 **已定**；武装类模板清单 **已定**，见下表；作业类模板资产 **待定**，sy-21）。

- **编制 ≠ 从住宅迁出**：创建队伍**占用**城市人口池中的编制名额，**不**把人员从来源城区**移除**；编组人员**仍占用**绑定城区**住宅容量**（见 [人口与迁移 · 人口与住宅](../04-资源与人口/人口与迁移.md#人口与住宅已定)）。
- **单一人口类型**：编组时锁定 `population_type_id`，**禁止**混编（见 [编组 · 单一人口类型](../04-资源与人口/人口与迁移.md#编组--单一人口类型已定)）。
- **城区来源人数**：编组时记录各城区来源编制数；**不**追踪个体；减员时**按权重随机**落点城区（见 [编组 · 城区来源人数](../04-资源与人口/人口与迁移.md#编组--城区来源人数已定)）。

| 项 | 结论 |
|----|------|
| **定义** | 面向**编队**的可调配 **物品 / 装备组**（归属 [物品层](../03-图层与地点/地图图层.md) 或城市库存，存储层级 **待定**），用以表达「这支队伍凭什么具备某类能力」 |
| **粒度** | 以**队伍实例**为单位：组建时消耗或占用；解散 / 撤回时是否返还 **待定** |
| **与四类资源** | **金属 / 能源 / 食物 / 人口** 为宏观资源；队伍资产为**模板级组建门槛**，不与「维持人口」「远征食物开关」混谈 |
| **与能力 SO** | 模板挂载的攻击 / 作业能力决定**能做什么**；队伍资产决定**能否组建该模板** |

### 默认编制与人数影响

> 默认值可在配置文件中调整，以下为当前设计。**默认人数即编制上限**（同一列）。

| 队伍类型 | 编制人数（默认 = 上限） |
|----------|-------------------------|
| 侦察队 | **3** |
| 运输队 | **15** |
| 工程队 | **10** |
| 民兵 | **15** |
| 步兵 | **10** |
| 弓手 | **10** |

- **队伍数量**：**无**全局可同时维持队伍数上限；实际可组建数量由城市**人口池**可划拨编制、[队伍资产](#队伍资产与组建门槛) 与模板编制人数 **共同**约束，**不**另设「最多 N 支队伍」硬顶。
- 队伍人数可因战斗或事件**低于满编**；**至少 1 人**即可维持在编（`current_headcount ≥ 1`）；降至 **0** 视为解散 / 失效（细则 **待定**）。
- **满编人数** = 模板编制人数（上表）；**人数比** = `current_headcount / 编制人数`（&lt; 1 时为减值）。创队**不要求**一次达到满编。

**人数对比率影响的能力通道（已定框架）**

| 受影响项 | 通道 | 说明 |
|----------|------|------|
| **视野范围** | 人数比系数 × 满编视野 | 人数越低，可揭示范围越小 |
| **负载上限** | 人数比系数 × 满编负载 | 运输与装卸上限随人数下降 |
| **工作效率** | **参与人数 × `work_efficiency`** | 人数**只**作为工作量乘数；见 [工作效率与工作量](../07-玩法循环/回合与行动表.md#工作效率与工作量) |
| **攻击效果 / 战力** | 人数比系数 × 满编战力；交战公式中 **人数系数** | 仅 **具备武装** 的模板生效；结算见 [战斗结算流程](#战斗结算流程当前版本) |

- **移动速度不**随人数比变化；由模板 `unit_base_profile.base_move_speed`（及 Buff 等其它修正）决定，**不**经人数比系数。
- 视野、负载、攻击/战力经程序表 **`headcount_factor_curve`** 统一映射：首版 **`linear`**，**系数 = 人数比**（满编 1.0，半编 0.5），写入 `ability_runtime_json`（程序见 [人数比影响系数](../../03-程序设计/03-数据字典/队伍与战斗数据结构.md#表-8headcount_factor_curve人数比影响系数)）。
- **工作效率**走 **`participant_count × work_efficiency`**，**不在** `work_efficiency` 内重复乘人数比。
- 各模板 **SO 默认参数**（满编视野格数、满编负载、满编战力、移速等具体数值）归入 [04-数值框架/](../04-数值框架/)（该目录未建，配置表搭建后填写，sy-03）。

详情与模板侧口径见 [单位类型与视野 · 默认编制与人数比影响](./单位类型与视野.md#默认编制与人数比影响)。

### 能力通道与参数配置

- 每项能力由独立能力配置 SO 定义，示例：
  - **建造能力**：二级选择设施类型；所需工作量对照值、资源消耗、按设施类型的建造条件（见 [单位类型与视野 · 工程队](./单位类型与视野.md#4-工程队模板)）；每回合增量受 [工作效率与工作量](../07-玩法循环/回合与行动表.md#工作效率与工作量) 影响；
  - **运维能力**：对已存在设施的运维；所需工作量与每回合增量受工作效率影响；
  - 搬运能力：装货/卸货为按回合推进的工作（所需工作量对照值各 1）；每回合增量受工作效率影响；负载上限；**不允许超载**——人数不足时按比例自动丢弃货物（见 [单位类型与视野 · 运输队](./单位类型与视野.md#3-运输队模板)）；
  - 攻击能力：打击范围、攻击效果、命中修正、反击许可。
- 队伍模板 SO 只负责“启用哪些能力”与“能力参数引用”，不在程序中写死。
- 程序层只提供能力执行入口（如 `TryBuild`、`TryTransport`、`TryAttack`），具体数值与开关完全读取 SO。

### 载荷、粮食与外派补给

与 [粮食与周总结 · 已定案详述](../../01-草稿/归档/粮食与周总结/粮食与周总结-已定案详述.md) 对齐：

| 项 | 规则 |
|----|------|
| **消耗** | 每周期 **1** 单位食物 / 编制人口 + **随行人员**（周总结口径 = 1 粮食/人/周） |
| **载荷目标** | 队伍默认载荷须能维持约 **3** 个周期的粮食（具体数值 **待定**，sy-21） |
| **长期外派** | 超出载荷续航时，须由 **运输队** 从城市向野外队伍输送物资（细则 **待定**，sy-21 / O-16） |
| **分池** | 城市仓库与队伍载荷**独立**结算，见粮食定案 §2.6 |

### 战斗身份与关系判定

交战参与方类型（队伍、设施、城市）与敌我规则见 [交战系统](./交战系统.md)。本节保留队伍侧摘要：

### 队伍 AI 接管规则（连续指令）

- 玩家可下达连续指令，形成有先后顺序的**指令表**（程序实现为指令队列；见 [回合与行动表 · 指令表与自主执行](../07-玩法循环/回合与行动表.md#指令表与自主执行)）。
- 指令表列出**任务目标**；**如何执行**由 AI 自主决定（路径、等待、遇敌应对等）。
- 队伍 AI 按表顺序逐步执行任务，不跳过任务目标类型，不擅自改写任务目标。
- 当前指令受阻时，AI 可在同一任务目标内做可行调整（换位、等待、改路径、改攻击点）。
- 若持续受阻且无法推进，AI 将该条标记为未完成并进入下一条，同时写入战报说明原因。
- [遭遇意外](../07-玩法循环/回合与行动表.md#遭遇意外的中断与恢复)时，不打断指令表结构，仅打断当前条目的自主执行；恢复后从同一条目继续。

### 任务优先级模型

- 指令队列以玩家下达顺序为第一优先级。
- 单条指令内部采用「任务目标 > 生存判断 > 执行动作」的决策顺序。
- 队伍 AI 策略可设为：
  - **保守**：优先降低损失，再推进任务目标。
  - **激进**：优先推进任务目标，再处理损失控制。

### 指令可行性预判

队伍在**执行每条指令前**，对理想情况下的关键数据进行预判，判断该指令是否具备执行条件。预判**不**代表实际执行中的保证（遇敌、意外中断等可能改变结果），仅作为AI是否推进指令的**前置检查**。

| 预判项 | 说明 |
|--------|------|
| **口粮续航** | 队伍载荷中的食物是否足以支撑**执行该指令并返回基地**（移动消耗的回合数 × 编制与随行人员口粮需求）；不足时标记「口粮不足」，AI 暂停推进该指令并写入战报 |
| **负载容量** | 执行指令预期的拾取 / 装载量是否超出队伍剩余负载上限；超出时按可容纳量执行（如寻回仅拾取可装下部分）或标记无法执行 |
| **路径可达** | 目标格是否可抵达（无不可通行地形阻断）；不可达时标记指令失败 |

- 预判在指令**开始执行时**运行一次；执行过程中若条件变化（如口粮在中途被消耗、负载因拾取变化），不重复预判，由 AI 接管规则在实际受阻时处理。
- 预判结果写入战报，玩家可在指令执行前查看「预计消耗口粮」「预计往返回合数」等摘要。

### 遭遇敌人时的行为

单位在移动或执行指令过程中**遭遇敌对单位**（进入威胁范围、路径被阻断、或被动卷入交战）时，**不采用全局固定反应**（如一律回避或一律交战），而由当前 **`ai_strategy`** 决定优先级：

| 策略 | 遇敌优先级 | 典型动作（在同一任务目标内调整） |
|------|------------|----------------------------------|
| **保守** | 降低损失 **先于** 推进任务 | 改道绕行、脱离接触、暂停等待威胁离开；仅在损失可控且能继续推进时才恢复当前指令 |
| **激进** | 推进任务 **先于** 损失控制 | 优先维持当前指令方向与目标；必要时交战或穿越威胁区以完成任务，再处理减员 |

约束：

- 遇敌反应**不改写**指令队列中的任务目标类型，仅在当前 `team_command_item` 内做可行调整（与 [队伍 AI 接管规则](#队伍-ai-接管规则连续指令) 一致）。
- **无武装**队伍（默认侦察队、运输队、工程队）不具备主动交战能力；激进策略下仍优先推进任务，但只能通过改路径、等待或接受风险通过等方式执行，**不得**凭空获得战斗能力。
- 若遇敌后持续无法按当前策略推进，按 AI 接管规则标记当前条目未完成或失败，并写入战报说明。

程序实现见 [队伍与指令执行 · 遇敌判定](../../03-程序设计/02-运行时逻辑/队伍与指令执行.md#遇敌判定与策略分支)。

### 队伍粮食与载荷（周总结）

队伍粮食在 **周总结** 结算，**不**每回合从城市池扣粮（**废止** OPEN-042 每回合路径与 `team_food_upkeep_enabled` 开关）。详见 [粮食与周总结（草稿）](../../01-草稿/归档/粮食与周总结/README.md)。

| 项 | 结论 |
|----|------|
| **分池** | **城市仓库**（按 `mobile_city_id`）与**各队载荷粮食**独立；编制 **不算** L1 驻留切分，**仍占**绑定城区住宅 |
| **需求** | 队伍容器 = **编制** + **随行人员**；各 1 粮食/人/周 |
| **不足** | 未分到 → **⌊50%⌋ 减员**；无饥饿中间态 |
| **随行人员** | 非编制、不参与队伍工作；占载荷、增粮食需求；接纳时已属玩家（见 [人口与迁移 · 随行人员](../04-资源与人口/人口与迁移.md#随行人员)） |
| **运输载具类** | **搬运队 / 运输队**（sy-21）主物资运输；与随行人员**分轨** |

程序占位：`TeamCargoState`、`EscortAccompanyingState`（见 [粮食与周总结 · 程序落地](../../01-草稿/归档/粮食与周总结/粮食与周总结-程序落地.md)）。

### 高级指令

玩家可为具备相应能力的队伍下达**高级指令**，扩展指令表的行为维度。高级指令与普通指令（移动、工作、攻击）使用同一套指令队列，遵守 [队伍 AI 接管规则](#队伍-ai-接管规则连续指令) 与 [任务优先级模型](#任务优先级模型)。

#### 追猎

**前提**：队伍须具备**攻击能力**（武装类模板）；目标为**已发现的敌方单位**或**已探查格内**的敌人。

| 项 | 规则 |
|----|------|
| **指令类型** | `team_cmd_hunt`（高级指令） |
| **目标** | 玩家指定一个**已知敌方单位**（须在己方视野或已探查格内） |
| **执行方式** | 队伍以目标单位为**追踪终点**，AI 自主规划接近路径；在追踪过程中若目标移动，动态更新路径 |
| **交战** | 进入打击范围后**自动发起攻击**（遵循 [遭遇敌人时的行为](#遭遇敌人时的行为)，激进策略优先交战）；战斗结算见 [战斗结算流程](#战斗结算流程当前版本) |
| **结束条件** | 目标被消灭；目标脱离可追踪范围（脱离视野且无法继续追踪）；或玩家手动移除指令 |

#### 封锁

**前提**：队伍须具备**攻击能力**（武装类模板）。

| 项 | 规则 |
|----|------|
| **指令类型** | `team_cmd_blockade`（高级指令） |
| **目标** | 玩家指定一个**已探查的格子**（可为资源点、关键路口、已发现敌人所在格等） |
| **执行方式** | 队伍**驻守**该格，进入待机响应状态：**主动袭击**进入该格或相邻格的**来袭敌方单位** |
| **响应范围** | 驻守格 + **相邻六格**（与移动邻接一致）；敌方单位进入此范围后，封锁队伍**主动**发起攻击 |
| **交战** | 自动进入战斗；遵循 [战斗结算流程](#战斗结算流程当前版本)；ai_strategy 在封锁中默认偏向**激进**（优先消灭来犯之敌） |
| **结束条件** | 玩家手动移除指令；队伍战力不足以继续驻守（人数降至 0 则解散，见 [单位死亡与物资掉落](#单位死亡与物资掉落)） |

#### 寻回

**前提**：队伍须具备**搬运能力**（默认模板中的运输队或拥有搬运能力的自定义模板）。

| 项 | 规则 |
|----|------|
| **指令类型** | `team_cmd_retrieve`（高级指令） |
| **目标** | 玩家指定**地图上的某一格**——该格存在**可拾取物资**（如 [单位死亡掉落](#单位死亡与物资掉落) 的物资堆、遗落的物品等） |
| **执行方式** | 队伍自动前往目标格 → **装货**（拾取该格全部可拾取物资）→ **自动返回基地**（出发时的归属城区或最近己方城区）→ **卸货**（将物资存入城区仓库节点） |
| **装货 / 卸货** | 各走 [工作中](../07-玩法循环/回合与行动表.md#工作中状态) 进度（所需工作量对照值各 **1**）；受 [工作效率与工作量](../07-玩法循环/回合与行动表.md#工作效率与工作量) 影响；超载规则与 [运输队模板](./单位类型与视野.md#3-运输队模板) 一致 |
| **遇敌行为** | 遵循当前 `ai_strategy`（保守 / 激进）；无武装运输队遇敌时按策略绕行或等待 |
| **结束条件** | 物资成功卸入城区仓库；目标格无可拾取物资（已被其他单位拾走）；负载已满无法装下时仅拾取可容纳部分后返回；或玩家手动移除指令 |

- **寻回** = 移动 + 装货 + 返回 + 卸货 的**一体化指令**，指令表中占**一条**；玩家不需手动拆成多步。
- 返回的**目标基地**默认取队伍**归属城区**（创队时绑定的城区）；若归属城区已不存在，取距离最近的**己方城区**。

### 快速建造（资源点）

> 本节为 [需求驱动指挥（工作计划）](../01-核心体验/平台与操作.md#需求驱动指挥工作计划) 体系的已有实例：先下达建造需求、再指派工程队（含新建）。

玩家在**已发现的资源点**上可以**快速发起建造需求**，无需先组建工程队再下达建造指令：

| 步骤 | 操作 |
|------|------|
| **1. 发起** | 在地图上选中**已揭示**的资源点（见 [资源点揭示](./单位类型与视野.md#资源点揭示)）→ 发起「建造设施」 |
| **2. 选设施** | 系统列出该资源点**可建造的设施类型**（如矿区、果园、能源站等）；玩家选择目标设施 |
| **3. 指派队伍** | 系统列出**可调用工程队**（含已编组工程队与可新建工程队的资源）；玩家**选取现有队伍**或**新建工程队**执行 |
| **4. 入队** | 确认后：所选工程队的指令表**追加**建造工作（`team_cmd_work_build`，目标 = 资源点格 + 设施类型）；若新建工程队则先触发 `form_team` 工作再追加建造 |

- **不改变** [工程队 · 建造设施](./单位类型与视野.md#4-工程队模板) 的二级选择流程与建造条件——快速建造仅缩短「找队伍 → 下指令」的操作路径，**不绕过**设施建造的前置条件与工作时长。
- 若当前无可调用工程队且无法新建（人口不足、无队伍资产等），提示玩家原因并列出缺失条件。
- 快速建造属于 [需求驱动指挥（工作计划）](../01-核心体验/平台与操作.md#需求驱动指挥工作计划) 的需求先行入口；亦可不当场指派，先保留为**工作计划**稍后指派（细则待定，sy-37）。

### 单位死亡与物资掉落

队伍在战斗中**人数降至 0**（`current_headcount = 0`）时解散。**人口死亡本身不掉落物资**；仅以下两项在队伍解散时留存在当前格：

| 项 | 规则 |
|----|------|
| **判定** | `current_headcount` 降至 **0** 时队伍**解散**（见 [默认编制与人数影响](#默认编制与人数影响)） |
| **人口** | 人口死亡**不掉落**任何物资 |
| **队伍资产** | 武装类模板的**队伍资产**（简陋兵甲、弓具等）**掉落**在队伍**当前所在格**；作业类模板队伍资产掉落规则同 |
| **掉落物存在形式** | 以**可拾取物品**的形式存在于该格（程序表现为该格的临时物品节点）；任何具备搬运能力的队伍可**拾取**（装货） |
| **时效** | 掉落物**不消失**（无自动清理计时器）；可被其他单位拾取、被环境事件影响或覆盖（细则 **待定**） |
| **随行人员** | 队伍解散时，随行人员视为**失散**于当前格（存活 / 损失比例 **待定**；见 [人口与迁移 · 随行人员](../04-资源与人口/人口与迁移.md#随行人员)） |

### 反击限制

- 仅当防守方具备反击能力，且攻击者位于防守方打击范围内，才触发反击。
- 反击按正常战斗流程结算。
- 反击不会触发新的反击响应（禁止反击连锁）。

### 战斗结算流程（当前版本）

完整交战管线见 [交战系统 · 战斗结算](./交战系统.md#战斗结算当前版本)。队伍间对攻：

1. 判定可攻击目标。
2. 发起攻击。
3. **造成伤害**（发起方）：攻击能力系数 × **人数系数** → 经防守方防御与人数系数 → 得到 **`damage`**（与人口同单位）。
4. **受到伤害**（遭受方）：遍历每条 Applicable 结构（减免 → 承伤，损伤累计 **1:1**）；**多条结构可同时保护同一人口**；全部结构处理后的 **剩余伤害** 才结算人口损失（见 [交战系统 · 受到伤害](./交战系统.md#受到伤害遭受方-已定)）。
5. 若满足反击条件，按同流程执行一次反击（**不含士气**）。

士气与逆风保护机制已废弃，见 [交战系统 · 已废弃机制](./交战系统.md#已废弃士气与逆风保护)。
