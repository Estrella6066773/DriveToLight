## 2. 已定案详述

### 2.1 周总结：时机与 Turn 挂钩


| 项   | 规则                                                                                     |
| --- | -------------------------------------------------------------------------------------- |
| 首次  | 第 **7** 回合：指挥 → 玩家行动 → AI 行动 → **环境行动** → **[周总结]**                                    |
| 后续  | 第 **14、21、28…** 回合环境行动后                                                                |
| 下回合 | 周总结 **全部子步骤完成** 后进入下一回合 **玩家指挥**                                                       |
| 判定  | `ShouldRunWeeklySummary(turn) => turn % 7 == 0`                                        |
| 调用  | `EnvironmentAction` 完成 → `WeeklySummaryOrchestrator.Run(ctx)` → 递增回合 → `PlayerCommand` |
| 存档  | `weekly_summary_index`（从 1 起；旧名 `summary_week_index` 可别名）                              |


```text
第 7 回合: 指挥 → 玩家行动 → AI 行动 → 环境行动
          → [周总结: Orchestrator 按模块顺序执行 IWeeklySummaryStep]
第 8 回合: 指挥 → …
```

### 2.1b 周总结编排架构（可重排、易扩展）

**设计目标**

- 子步骤 **按游戏模块分块**（Resource、Work、Buff、Map…），**不在 Turn 里写死** switch。
- **调整顺序** = 改 `WeeklySummaryStepCatalog` 注册/排序，**不**改 Orchestrator 核心逻辑。
- 每步 **可单测**；后期可加 ScriptableObject 配置（**可选**）。

**程序集**

- 编排壳：`**Yanxu.Turn**`（或 `Yanxu.Core`）— `Orchestrator`、`Catalog`、接口。
- 步骤实现：落在 **所属模块**（Resource、Map…）；Turn **只依赖接口**。

```text
Yanxu.Turn
├── WeeklySummaryModule
├── IWeeklySummaryStep
├── WeeklySummaryContext
├── WeeklySummaryStepCatalog     // Register + GetOrderedSteps()
└── WeeklySummaryOrchestrator

Yanxu.Resource
├── ResolveMobileCityGroupsStep  // 或合入下一步
├── PopulationFoodWeeklySummaryStep
└── FoodAllocator / cohort / WeeklyPopulationFoodPolicy

其它模块
├── BuffRefreshWeeklySummaryStep
└── MapRefreshWeeklySummaryStep
```

**核心类型（草案）**

```csharp
enum WeeklySummaryModule
{
    Resource = 100,
    Work = 150,
    Buff = 200,
    Map = 300,
    Sun = 400,
}

interface IWeeklySummaryStep
{
    string StepId { get; }
    WeeklySummaryModule Module { get; }
    int OrderInModule { get; }
    void Execute(ref WeeklySummaryContext context);
}

readonly struct WeeklySummaryContext
{
    public int TurnNumber;
    public int WeeklySummaryIndex;
    public IReadOnlyList<string> MobileCityIds;
}

sealed class WeeklySummaryStepCatalog
{
    void Register(IWeeklySummaryStep step);
    IReadOnlyList<IWeeklySummaryStep> GetOrderedSteps();
    // 排序：Module 升序 → OrderInModule → StepId
}

sealed class WeeklySummaryOrchestrator
{
    WeeklySummaryRunResult Run(in WeeklySummaryContext context);
}
```

**首版默认步骤（定案：按模块；模块内顺序在 Catalog 可调）**


| 序   | Module   | StepId                       | 职责                                                      |
| --- | -------- | ---------------------------- | ------------------------------------------------------- |
| 1   | Resource | `resolve_mobile_city_groups` | 解析相连分区 → 写入 Context                                     |
| 2   | Resource | `population_food_settlement` | 逐 `mobile_city_id`：汇总库存 → `FoodAllocator` → cohort → 扣减 |
| 3   | Work     | `apply_work_headcount_delta` | cohort 变化 → `ApplyWorkHeadcountChange` / 队伍等价 → GAS 投影  |
| 4   | Buff     | `buff_refresh`               | buff 再发放（首版可 NoOp）                                      |
| 5   | Map      | `map_refresh`                | 地图状况刷新（首版可 NoOp）                                        |


> **口粮算法内部**（步骤 2 内）保持：汇总 → 分配（在岗优先）→ cohort → 扣减。若日后须 **buff 先于口粮**，只改 Catalog 的 Module/Order，**不动** `FoodAllocator`。

**扩展约定**

- 新系统：新类实现 `IWeeklySummaryStep` + `Register`。
- 步骤尽量 **幂等** / **可重跑**（读档、单测）。
- Turn 阶段机 **仅**：`if (ShouldRunWeeklySummary) orchestrator.Run(ctx)`。
- `[PopulationFoodSettlementService](../../../Assets/Scripts/Yanxu.Resource/Population/Food/PopulationFoodSettlementService.cs)` 由 `PopulationFoodWeeklySummaryStep` 调用；`[WeeklyPopulationFoodPolicy](../../../Assets/Scripts/Yanxu.Resource/Population/Food/)` 承载口粮算法。

### 2.2 移动城市分区与玩家经营范围

**术语**


| 用语     | 含义                                   |
| ------ | ------------------------------------ |
| **相邻** | 地图格物理临近；**不**等于同一移动城市                |
| **相连** | 经城区 **连接拓扑** 与 **核心区网络** 连通（地图上可有距离） |


`**mobile_city_id` 划分**

- **相连** 连通分量中，凡能连到 **至少一个核心区** 的城区 = **1 座移动城市**。
- 多核两半 **仍拓扑相连** → **1** 座；**断成两个各含核心的分量** → **2** 座，口粮 **各自独立**。
- **分离** 后不能连到任何核心区 → **不属于** 移动城市（落地图独立实例；口粮账见 **O-08**）。

**势力主城区 vs 玩家经营范围**

- **势力主城区**（领袖任职城区）定义 **不变**（见 [领袖与势力](../../02-系统设计/05-城市与领袖/领袖与势力.md)）。
- **玩家可经营 / 玩家口粮账** = **领袖已入玩家阵营** 的城区及其 **相连** 网络，**≠** 势力主城区全集，**≠** 军事占领区。

**玩家 vs 占领**


| 状态                   | 口粮账 / 周总结                           | 城市管理系统 |
| -------------------- | ----------------------------------- | ------ |
| 领袖 **已入玩家阵营**（含相连网络） | 玩家；进玩家 `mobile_city_id`             | 可用     |
| **仅占领、领袖未入阵营**       | **不** 归玩家；**不** 进玩家 `FoodAllocator` | 不可用    |
| 非玩家外部城市              | AI 抽象统管；同规则周总结                      | 不开放    |


> **文档冲突**：[城市管理系统](../../02-系统设计/04-资源与人口/城市管理系统.md) 现稿「占领即可经营」**废止**，以本定案为准。

### 2.3 人口两层：活跃容器 vs 村镇储量


| 层         | 对象                     | 周总结                     |
| --------- | ---------------------- | ----------------------- |
| **可提取储量** | 村镇 `PopulationReserve` | **不占用** 口粮，直到 **提取**    |
| **活跃人口**  | 城区、队伍                  | **1:1** 需求、饥饿/致死 cohort |


**活跃人口容器**


| 容器     | 说明                                               |
| ------ | ------------------------------------------------ |
| **城区** | 居民与上岗；**废墟** 计口粮、不可工作、**不可迁入**、居民 **仅可迁出**       |
| **队伍** | 编制 + 载荷迁徙人口（**共享口粮**）；**出征前应预先装入载荷口粮**；载荷亦可装其它物资 |


**迁移**


| 方式      | 条件                                    |
| ------- | ------------------------------------- |
| 同移动城市   | **相连** + 同一 `mobile_city_id` + 玩家经营城区 |
| 跨分区     | 分离/再连接，或运输队载荷                         |
| 村镇 → 势力 | 提取 → 城区或队伍（OPEN-033 细则待定）             |
| AI      | 与玩家 **同一套** 逻辑                        |


### 2.4 口粮：需求、分配、饥饿与致死

**需求与库存**

- 需求 = 分区内活跃人口 × **1** 口粮/人/周。
- 可用 = 分区内城市 `ItemInventory` 口粮 + 绑定该 `mobile_city_id` 的队伍 **载荷口粮**。
- 跨 `mobile_city_id` **不** 自动均分；调拨靠运输/贸易（细则待定）。

**分配（已定：两种不足算法 + 可切换）**


| 情况          | 行为                                                                                |
| ----------- | --------------------------------------------------------------------------------- |
| **口粮 ≥ 需求** | 人人 1:1 满足；不跑不足算法                                                                  |
| **口粮 < 需求** | 读 `FoodAllocationConfig.ShortagePriorityMode` → 调用对应 `**IFoodShortageAllocator`** |


**配置与扩展**

```csharp
enum FoodShortagePriorityMode
{
    WorkFirst = 0,           // 默认：工作者优先
    AffiliationFirst = 1,    // 归属（势力）优先
}

class FoodAllocationConfig
{
    FoodShortagePriorityMode ShortagePriorityMode = WorkFirst;
    AffiliationPriorityList AffiliationOrder;  // AffiliationFirst 模式用；WorkFirst 模式下仅用于余粮细分（待定）
    // 按 mobile_city_id 存盘；城市管理系统 UI 可改（OPEN-046）
}

interface IFoodAllocationConfigModifier
{
    void Apply(ref FoodAllocationConfig config, in FoodAllocationContext ctx);
}
// FoodAllocator 运行前：config = base; foreach (modifier) modifier.Apply(ref config, ctx);
// 领袖能力、buff、事件等模块后续注册 Modifier，不改 Allocator 本体
```

`**FoodAllocator` 结构**

```csharp
interface IFoodShortageAllocator
{
    FoodShortagePriorityMode Mode { get; }
    void AllocateShortage(in FoodAllocationContext ctx, ref int foodStock, List<FedResult> results);
}

class FoodAllocator
{
    IFoodShortageAllocator workFirst;      // WorkFirstFoodAllocator
    IFoodShortageAllocator affiliationFirst;

    void Run(mobileCityId, config, stock) {
        ResolveModifiers(ref config);
        if (stock >= totalDemand) { /* 全员 Fed */ return; }
        var algo = config.ShortagePriorityMode == WorkFirst ? workFirst : affiliationFirst;
        algo.AllocateShortage(ctx, ref stock, results);
        UpdateCohortFrom(results);  // 与 O-03 共用 SortKey / Fed 集合
    }
}
```

### 2.4a 算法 A：`WorkFirstFoodAllocator`（**默认**）

**语义**：口粮不足时 **最高优先级 = 正在工作者**（城区上岗 + 队伍进行中指令；载荷人口 **待定** O-07）。

```text
输入：DemandEntry[]（每人/每编制槽：container, affiliation, isOnDuty）
      foodStock

1. ON_DUTY = 所有 isOnDuty == true（城区 + 队伍合并，不先分归属）
2. 若 stock >= |ON_DUTY|：
     先喂满 ON_DUTY（各 1 口粮）
     stock -= |ON_DUTY|
     余粮按 **容器人数占比** 喂 OFF_DUTY（整数取整规则见 O-03）
   否则：
     stock 只分给 ON_DUTY：
     按 **全局在岗人数占比** 或 **稳定 SortKey** 分配整数口粮（与 O-03 一致）
     OFF_DUTY 全部未分到 → 饥饿
3. 输出 FedResult / Unfed → cohort
```

**玩家体验**：保运转、保远征；闲人、低活跃容器先饿。

### 2.4b 算法 B：`AffiliationFirstFoodAllocator`

**语义**：口粮不足时 **最高优先级 = 归属（势力）排序**；**同一归属内** 再 **工作者优先**。

```text
1. 按 AffiliationOrder 将 DemandEntry 分成归属桶
2. 对每个归属 G（高→低）：
     需求_G = |G|
     分配_G = min(stock, 需求_G)   // 或按配置权重切份额（首版可 min 顺序灌满）
     stock -= 分配_G
     在 G 内：
       先喂 isOnDuty，再喂非在岗（同 2.4a 的内层规则）
3. 低优先级归属在 stock 耗尽后整桶未分到 → 饥饿
```

**玩家体验**：保主力文化；低优先级归属的在岗者也可能后于高归属的非在岗者（若该归属份额已用尽）。

### 2.4c 与 O-03、UI 的衔接

- **O-02 已定**：两算法分文件实现；**默认 `WorkFirst`**；`FoodAllocationConfig` + `**IFoodAllocationConfigModifier**` 供它模块改写。
- **O-03 仍须定**：整数口粮 **取整 / tie-break**（两算法共用同一 `AllocationSortKey`  helper，保证「分到口粮 ↔ 解除饥饿 ↔ 致死选取」一致）。
- **O-14**：AI 默认 `**WorkFirst`**（与玩家一致），除非事件/Modifier 改写。

**饥饿两阶**


| 周次          | 条件     | 结果                |
| ----------- | ------ | ----------------- |
| 第 1 次未分到    | 新饥饿    | 标记饥饿；无法工作         |
| 第 2 次+ 仍未分到 | 上周期已饥饿 | 死 **⌊n/2⌋**；余下仍饥饿 |


```text
deaths = hungryCount / 2   // 向下取整
hungryCount -= deaths
TryRemovePopulation(deaths, FoodShortageDecay)
ApplyWorkHeadcountChange(...)
```

- 连续断粮：100 → 死 50 剩 50 → 死 25 剩 25 …（非一次清零）。

**OPEN-042 废止**

- 队伍 **不再** 每回合单独扣粮；与城区 **同一次** 周总结 `FoodAllocator`。
- `team_food_upkeep_enabled = false`（若保留）：队伍 **不计需求、不参与分配**；城区不变。

### 2.5 城区工作与队伍：ApplyWorkHeadcountChange


| 项     | 定案                                                                          |
| ----- | --------------------------------------------------------------------------- |
| 工作口径  | 城区运作 / 城内工作 / 设施运行 **同一件事**                                                 |
| 人力    | **仅本城区居民**；每城区 **一种归属** 上岗                                                  |
| 特殊城区  | **无单独逻辑**；经 `ApplyWorkHeadcountChange`：减员→效率变；达 `canSustainWork` 则 **能力不停** |
| 工作区关闭 | **手动** `WorkZoneEnabled`；**不经过** Delta                                      |
| 饥饿    | **被动**进 Delta；口粮恢复 **自动**解除                                                 |
| 废墟    | `GetWorkableResidents` **恒 0**                                              |
| 队伍    | 同 1:1 / cohort / 在岗优先；`GetWorkableTeamHeadcount` = 编制 − 饥饿（载荷人口 **待定**）     |


**Delta 输出（写入数据字典）**


| 输入    | 行为                         |
| ----- | -------------------------- |
| 饥饿/恢复 | 更新可工作人数                    |
| 战斗减员  | 同上                         |
| 不能工作者 | 自动撤岗                       |
| 维持门槛  | `canSustainWork`；false 才停摆 |


**模块边界**：`canSustainWork`、进行中工作响应、Delta 约定归属 **工作×人口**；城区层只提供输入与 SO 参数。

**程序约定**

- 链路：**cohort → `ApplyWorkHeadcountChange` → GAS 投影**；**禁止**在 `Yanxu.City` 内散落饥饿/维持分支。
- 进行中城内工作：人数变化后倾向 **完成**（OPEN-032）；由 Delta 或 `**WorkProgressService`** 衔接。
- 致死后人口裁剪：扩展 `[CityPopulationService.TrimResidentsAndTeamsAfterLoss](../../../Assets/Scripts/Yanxu.Resource/Population/CityPopulationService.cs)`（及 cohort 同步）。
