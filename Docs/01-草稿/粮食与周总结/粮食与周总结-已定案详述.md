## 2. 已定案详述

### 2.1 周总结：时机与 Turn 挂钩

| 项 | 规则 |
| --- | --- |
| 首次 | 第 **7** 回合：**全程正常行动**（指挥 → 玩家行动 → AI 行动 → 环境行动）→ **环境行动结束后** **[周总结]** |
| 后续 | 第 **14、21、28…** 回合环境行动后 |
| 下回合 | 周总结 **全部子步骤完成** 后进入下一回合 **玩家指挥**（如第 7 回合周总结后 → 第 8 回合指挥） |
| 判定 | `ShouldRunWeeklySummary(turn) => turn % 7 == 0` |
| 调用 | `EnvironmentAction` 完成（含周总结子步）→ `RelationshipAction`（关系结算）→ 递增回合 → `PlayerCommand` |
| 存档 | `weekly_summary_index`（从 1 起；旧名 `summary_week_index` 可别名） |

```text
第 7 回合: 指挥 → 玩家行动 → AI 行动 → 环境行动（仍可运粮、调仓、改编制）
          → [周总结: Orchestrator 按模块顺序执行 IWeeklySummaryStep]
          → 关系结算（脱离 / 效忠等）
第 8 回合: 指挥 → …
```

### 2.1b 周总结编排架构（可重排、易扩展）

**设计目标**

- 子步骤 **按游戏模块分块**（Resource、Work、Buff、Map…），**不在 Turn 里写死** switch。
- **调整顺序** = 改 `WeeklySummaryStepCatalog` 注册/排序，**不**改 Orchestrator 核心逻辑。
- 每步 **可单测**；后期可加 ScriptableObject 配置（**可选**）。

**程序集**

- 编排壳：`Yanxu.Turn`（或 `Yanxu.Core`）— `Orchestrator`、`Catalog`、接口。
- 步骤实现：落在 **所属模块**（Resource、Map…）；Turn **只依赖接口**。

```text
Yanxu.Turn
├── WeeklySummaryModule
├── IWeeklySummaryStep
├── WeeklySummaryContext
├── WeeklySummaryStepCatalog
└── WeeklySummaryOrchestrator

Yanxu.Resource
├── ResolveMobileCityGroupsStep
├── PopulationFoodWeeklySummaryStep
└── FoodAllocator / FoodWarehouseDrawdown / WeeklyPopulationFoodPolicy

其它模块
├── BuffRefreshWeeklySummaryStep
└── MapRefreshWeeklySummaryStep
```

**首版默认步骤（定案：按模块；模块内顺序在 Catalog 可调）**

| 序 | Module | StepId | 职责 |
| --- | -------- | ---------------------------- | ------------------------------------------------------- |
| 1 | Resource | `resolve_mobile_city_groups` | 解析相连分区 → 写入 Context |
| 2 | Resource | `population_food_settlement` | 逐 `mobile_city_id`：§2.8 扣减 → Override → L1→L2→L3 → 减员 |
| 3 | Work | `apply_work_headcount_delta` | 粮食/战斗减员 → `ApplyWorkHeadcountChange` |
| 4 | Buff | `buff_refresh` | buff 再发放（首版可 NoOp） |
| 5 | Map | `map_refresh` | 地图状况刷新（首版可 NoOp） |

> **粮食算法内部**（步骤 2 内）：`FoodWarehouseDrawdown` → `RunFoodOverrides` → `DefaultProportional` → `ApplyDeaths`。若日后须 **buff 先于粮食**，只改 Catalog 的 Module/Order。

**扩展约定**

- 新系统：新类实现 `IWeeklySummaryStep` + `Register`。
- Turn 阶段机 **仅**：`if (ShouldRunWeeklySummary) orchestrator.Run(ctx)`。
- `[PopulationFoodSettlementService](../../../Assets/Scripts/Yanxu.Resource/Population/Food/PopulationFoodSettlementService.cs)` 由 `PopulationFoodWeeklySummaryStep` 调用。

### 2.2 移动城市分区与玩家经营范围

**术语**

| 用语 | 含义 |
| ------ | ------------------------------------ |
| **相邻** | 地图格物理临近；**不**等于同一移动城市 |
| **相连** | 经城区 **连接拓扑** 与 **核心区网络** 连通 |

**`mobile_city_id` 划分**

- **相连** 连通分量中，凡能连到 **至少一个核心区** 的城区 = **1 座移动城市**。
- 多核两半 **仍拓扑相连** → **1** 座；**断成两个各含核心的分量** → **2** 座，粮食 **各自独立**。
- **分离** 后不能连到任何核心区 → **不属于** 移动城市（落地图独立实例；粮食账见 **O-08**）。

**玩家 vs 占领**

| 状态 | 粮食账 / 周总结 | 城市管理系统 |
| -------------------- | ----------------------------------- | ------ |
| 玩家**自有**城区 | 玩家；进玩家 `mobile_city_id` | 可用 |
| 招募 · **效忠**后原外部城城区 | **无**独立外部城主体；并入玩家 `mobile_city_id`（见 **§2.2.2**） | 可用（与玩家自有城区一致） |
| 招募 · **未效忠**（外部城） | **独立**结算主体；**不**并入玩家 `mobile_city_id`；扣粮见 **§2.2.1** | 有限（见 [势力城区管辖](../../02-系统设计/05-城市与领袖/领袖与势力.md#招募后势力城区管辖已定)） |
| **仅占领、领袖未入阵营** | **不** 归玩家（**O-09** 待决细则） | 不可用 |
| 非玩家外部城市 | AI 抽象统管；同规则周总结 | 不开放 |

> [城市管理系统](../../02-系统设计/04-资源与人口/城市管理系统.md) 现稿「占领即可经营」**废止**，以本定案为准。

#### 2.2.1 未效忠外部城 · 粮食扣减（已定）

**适用**：玩家已成功招募某**外部城市领袖**、**未效忠**期间。与 [领袖与势力 · 未效忠粮食](../../02-系统设计/05-城市与领袖/领袖与势力.md#未效忠粮食已定) 一致。

| 项 | 规则 |
|----|------|
| **结算主体** | 该外部城**独立**跑 `population_food_settlement`；活跃人口需求与其它城区相同（§2.3～§2.4） |
| **优先玩家** | **可**从玩家侧扣粮 **且** 玩家侧**足以**覆盖本城当周**全部**需求 → **仅**扣玩家相连分区的 §2.8 仓库 |
| **回退封存** | **无法**消耗玩家粮食 → 扣该城**封存粮食**。例：玩家粮尽；移动城市与本城**未相连** |
| **部分不足** | 相连但 §2.8 后仍不足 → 先扣尽玩家可扣部分，**剩余**从封存扣 |
| **仍不足** | 玩家侧 + 封存合计不足 → §2.4.5 **半数减员** |
| **封存边界** | 仅**粮食**可经回退路径动用；其它封存资源**不**参与周总结扣减 |
| **玩家操控** | **禁止**管理面板调拨；本路径**仅**周总结**自动**扣减（见 [未效忠资源管控](../../02-系统设计/05-城市与领袖/领袖与势力.md#未效忠资源管控已定)） |

**「可从玩家侧扣粮」**：本城经 [连接拓扑](../../02-系统设计/03-图层与地点/建筑层/连接与多核心.md) 与玩家**至少一处** `mobile_city_id` **相连**。

#### 2.2.2 效忠后 · 粮食与账主体（已定）

**适用**：玩家已成功招募某**外部城市领袖**且已 [效忠](../../02-系统设计/05-城市与领袖/领袖与势力.md#效忠已定)。

| 项 | 规则 |
|----|------|
| **外部城实体** | **消解**——**不再**作为独立 recruited 外部城跑账（见 [效忠 · 城市消解](../../02-系统设计/05-城市与领袖/领袖与势力.md#效忠城市消解与人口留存已定)） |
| **留存语义** | **城市领袖** + 人口**人口归属**（身份标签）；**不**保留独立城市经营主体 |
| **封存粮食** | 解封并入**玩家**仓库 |
| **周总结** | 原城活跃人口纳入玩家 **`mobile_city_id`** 连通分量；与玩家自有城区**同一** §2.8 → Override → L1→L2→L3 管线 |
| **§2.2.1** | **废止**适用；**不**再独立 `population_food_settlement` |

- 拓扑上若原城区与玩家移动城市**相连** → 并入**同一** `mobile_city_id`；**未相连**的分量仍按连通规则各自结算，但主体均为**玩家**（**非**外部城）。

### 2.3 人口两层：活跃容器 vs 村镇储量

| 层 | 对象 | 周总结 |
| --------- | ---------------------- | ----------------------- |
| **可提取储量** | 村镇 `PopulationReserve` | **不占用** 粮食，直到 **提取** |
| **活跃人口** | 城区、队伍（编制 + 随行人员） | **1:1** 需求；未分到 **半数减员**（§2.4.5） |

**活跃人口容器**

| 容器 | 说明 |
| ------ | ------------------------------------------------ |
| **城区** | **有住宅安置**在该城区的人口（**含**编组在外、仍占本城区住宅者；UI **居民数**）；**L1 粮食切分**时 **排除**已走队伍载荷的**编制**（§2.6）；**废墟** 计粮、不可工作、**不可迁入**、**仅可迁出** |
| **队伍** | **编制** + **随行人员**（§2.10）；编制 **恒属队伍容器**；粮食来自 **本队载荷** |

**不存在** `UnassignedPool` / 城市虚池；人口 **只** 在活跃容器或村镇储量。

**迁移**

| 方式 | 条件 |
| ------- | ------------------------------------- |
| 同移动城市 | **相连** + 同一 `mobile_city_id` + 玩家经营城区 |
| 跨分区 | 分离/再连接，或运输队载荷 |
| 村镇 → 势力 | 提取 → 城区或 **随行人员**（OPEN-033 细则待定） |
| AI | 与玩家 **同一套** 逻辑 |

### 2.4 粮食：分配、减员与周总结步骤

#### 2.4.1 分配单元与分池

在 **单个 `mobile_city_id`** 内，**城市粮食** 与 **队伍粮食** **分池、分容器**，不合并后再按城区人口切分。

1. **城市侧**：按 **§2.8 仓库扣减** 从各 **食物仓库节点** 实扣库存
2. **城市侧**：**按各城区当前驻留人口占比** 切分到各城区（**不含** 城外队伍编制，§2.6）
3. **各城区**：按 **§2.4.2** L2 归属占比 → L3 归属内同比例
4. **队伍侧**：各队 **载荷粮食** 在 **队伍容器** 内分配（**编制** + **随行人员**，§2.10）
5. **未分到者**：**死亡数 = ⌊未分到人数 × 50%⌋**；余者存活、无中间状态（§2.4.5）

#### 2.4.2 常规分配（L1→L2→L3）

| 层 | 规则 |
|----|------|
| **L1 · 分区内城区** | 城市粮食 → 各城区份额 = 本城区驻留人口 / 分区内城区驻留人口合计 |
| **L2 · 城区内归属** | 城区份额 → 归属 \(a\) 子份额 = 归属 \(a\) 人口 / **本城区总人口** |
| **L3 · 归属内个体** | 子份额内 **同食物/人口比例**；**不因是否在岗** 改比例 |

| 项 | 规则 |
|----|------|
| **充足** | 人人 1 粮食/人/周 |
| **不足** | 归属内 \(r = \min(1,\ \text{子份额粮食}/\text{该归属人口})\)；离散整数见 **§2.4.6**；未分到减员见 **§2.4.5** |
| **废墟** | 计需求；未分到 **同样半数减员**；**仍不能工作** |
| **连续断粮** | 下周期对 **新的未分到集合** 再 **半数减员** |

**示例**：城区 A 分到 100 粮食；人口 200（甲 120、乙 80）→ 甲 60、乙 40。

> **废止**：全局 `WorkFirst` / `AffiliationFirst` / `FoodShortagePriorityMode`（见 §2.4.3）。

#### 2.4.3 势力 / 领袖 / 人口归属能力（Override）

| 项 | 规则 |
|----|------|
| **来源** | 注册 **`IFoodAllocationRuleOverride`** |
| **定位** | **仅优先执行一轮**；已结算人口 **不再进入** L1～L3 |
| **粮食** | 能力修改的粮食在一般规则开始前已反映于 `cityStock` / 队伍载荷 |
| **无串流** | **§2.8 扣减** → **Override** → **一般 L1→L2→L3**；顺序单管线 |

```text
stock = FoodWarehouseDrawdown.TotalDrawn
(resultsOverride, stock) = RunOverrides(ctx, stock, population)
resultsDefault = DefaultProportional.Run(ctx, stock, remainingPopulation)
merge resultsOverride + resultsDefault
```

**多 Override**：按 **优先级 / 注册序** 依次执行（程序定稿）。

#### 2.4.4 新版「工作者优先」能力（仅 Override）

| 项 | 规则 |
|----|------|
| **形态** | **`IFoodAllocationRuleOverride`**（领袖或人口归属能力）；**非** 全局策略 |
| **默认** | 无此能力时：在岗者与同城区其他归属 **同等** 比例均分 |
| **能力语义** | ① 先为 **参与工作** 人口配给（城区上岗、队伍执行中 **编制**）；② **剩余** stock 对剩余人口走 L2→L3 |
| **参与工作** | **不含** 随行人员（§2.10） |

#### 2.4.5 未分到减员

| 项 | 规则 |
|----|------|
| **致死** | `DeathCount = ⌊UnfedCount / 2⌋` |
| **存活** | `SurvivingUnfedCount = ⌈UnfedCount / 2⌉` |
| **执行** | Unfed 子集内按 **§2.4.6 同一 SortKey**，取前 `DeathCount` 人 `TryRemovePopulation(..., FoodShortageDecay)` |
| **状态** | 幸存者 **本周期可工作**（废墟除外） |
| **示例** | 5 未分到 → 死 2、活 3 |

#### 2.4.6 整数粮食与 AllocationSortKey

| 步 | 规则 |
|----|------|
| **排序** | `AllocationSortKey` **升序**（稳定排序；tie-break **程序待定**） |
| **分到** | 整数粮食 `F`；按序每人至多 1 单位 → **Fed** / **Unfed** |
| **减员** | Unfed 子集同一顺序 → §2.4.5 |

**SortKey 字段（程序待补）**：`PopulationUnitId` 或 `(LocationRef, AffiliationId, RegisterIndex)` 等确定性键。

#### 2.4.7 周总结 Resource 步骤伪代码

```text
foreach mobile_city_id:
  draw = FoodWarehouseDrawdown.Run(mobile_city_id, demand)
  cityStock = draw.TotalDrawn
  (overrideResults, cityStock) = RunFoodOverrides(ctx, cityStock, cityPopulation)
  cityResults = FoodAllocator.RunDistricts(ctx, cityStock, remainingPopulation)
  foreach boundTeam:
    teamStock = team.CargoFood
    (teamOverride, teamStock) = RunFoodOverrides(ctx, teamStock, teamPopulation)
    teamResults = FoodAllocator.RunTeam(ctx, teamStock, remainingTeamPopulation)
  merge results → ApplyDeaths → ApplyWorkHeadcountChange
```

**OPEN-042 废止**：队伍 **不再** 每回合单独扣粮；编入周总结。

### 2.5 城区工作与队伍：ApplyWorkHeadcountChange

| 项 | 定案 |
| ----- | --------------------------------------------------------------------------- |
| 工作口径 | 城区运作 / 城内工作 / 设施运行 **同一件事** |
| 人力 | **仅本城区居民**；每城区 **一种归属** 上岗 |
| 特殊城区 | **无单独逻辑**；经 Delta：减员→效率变；达 `canSustainWork` 则 **能力不停** |
| 工作区关闭 | **手动** `WorkZoneEnabled`；**不经过** Delta |
| 粮食减员 | **被动**进 Delta |
| 废墟 | `GetWorkableResidents` **恒 0** |
| 队伍 | 编制参与工作；`GetWorkableTeamHeadcount` = 可工作编制（**不含** 随行人员） |

**Delta 输入**

| 输入 | 行为 |
| ----- | -------------------------- |
| 粮食减员 | 更新可工作人数 |
| 战斗减员 | 同上 |
| 不能工作者 | 自动撤岗 |
| 维持门槛 | `canSustainWork`；false 才停摆 |

**程序约定**

- 链路：**减员结果 → `ApplyWorkHeadcountChange`**；**禁止**在 `Yanxu.City` 内散落粮食维持分支。
- **队伍编制减员**：须按 [编组 · 城区来源人数](../../02-系统设计/04-资源与人口/人口与迁移.md#编组--城区来源人数已定) 更新 `district_headcount_json` 并落点城区居民数。
- 进行中城内工作：人数变化后倾向 **完成**（OPEN-032）。
- 致死后裁剪：`CityPopulationService.TrimResidentsAndTeamsAfterLoss`。

### 2.6 队伍与城区：人口口径分离

| 项 | 规则 |
|----|------|
| **编制 · 指挥** | **恒属队伍容器**（出征形态）；**不算** 城区 L1「城市仓库切分」人口 |
| **编制 · 住宅** | **仍归属**绑定城区；**仍占用**该城区 **住宅容量**；创建队伍 **不**从城区移除居民 |
| **城区 · 居民数** | UI 与承载校验 = **有住宅安置**者（**含**编组在外） |
| **城区切分（L1）** | **仅**从 **城市仓库** 吃粮的城区人口；**不含**走队伍载荷的编制 |
| **队伍切分** | **编制 + 随行人员**；粮食 = **载荷粮食**；**不参与** 城市 L1 |
| **载荷续航** | 默认载荷设计目标约 **3** 周期粮食（具体数值待定） |
| **长期外派** | 载荷不足时须 **运输队** 从城市输送物资（细则待定，O-16 / OPEN-044） |
| **减员** | 队伍未分到者 **⌊50%⌋ 减员**；整队 L3 同比例 |

**现有代码偏差**：`CityPopulationPoolState` 须拆为 **活跃分容器** + **村镇储量**。

### 2.8 仓库扣减：优先用于充饥

周总结 **城市侧** 从 **食物仓库**（城区 / 仓储设施 / `cargo_node` 等，程序统一 `FoodStorageNode`）扣减，规则来自 [城市管理系统 · 资源存储分配](../../02-系统设计/04-资源与人口/城市管理系统.md#资源存储分配) 中 **「优先用于充饥」** 勾选。

| 项 | 规则 |
|----|------|
| **结算范围** | 每 `mobile_city_id` **独立** |
| **队伍载荷** | **不走** 本节 |
| **优先仓** | 勾选「优先用于充饥」的节点 |
| **阶段 1** | 有优先仓 → 需求 `R` 在优先仓间 **均分** 扣减 |
| **阶段 2** | 仍缺 → 在 **其它仓** 间均分 |
| **无优先仓** | 直接在 **全部食物仓库** 间均分 |
| **单仓取尽** | 未满足份额 **重均分** 给同阶段有余量仓；无余量 → **食物用尽** |

```text
DrawFromTier(activeWarehouses, R):
  while R > 0 and active has stock:
    split R equally among warehouses with stock
    for each w: take = min(w.stock, w.share); w.stock -= take; R -= take
    remove depleted; redistribute unfulfilled share
  return R
```

**跨文档**：OPEN-046（勾选 UI）。

### 2.9 粮食充足性常驻 UI

**废止**旧「仅第 6 回合预告」；**每回合** 轻量判断与标注。

| 项 | 规则 |
|----|------|
| **形态** | **常驻 UI**（HUD / 地图角标等，线框待定） |
| **刷新** | **每回合**（指挥阶段入口 + 库存/人口变动后） |
| **判断单元** | 按 `mobile_city_id`：可读仓库 + 队伍载荷 vs 活跃人口需求 |
| **口径** | `supply >= demand` → 充足；否则不足（首版不强制完整 dry-run） |
| **多分城** | **仅对不足分区** 标注移动城市 |
| **与结算** | **只读**；实结算在周总结 |

```text
foreach mobile_city_id in playerScope:
  supply = SumFoodWarehouses(id) + SumTeamCargoFood(id)
  demand = ActivePopulationDemand(id)  // 含编制 + 随行人员
  if supply < demand → MarkMobileCityOnMap(id)
```

**程序占位**：`FoodSufficiencyIndicator` / `EvaluateMobileCityFoodStatus`。

### 2.10 随行人员（废止「迁徙人口」）

> **废止**：「载荷迁徙人口 / 迁徙人口」；功能由 **随行人员** 完整替代。

| 项 | 规则 |
|----|------|
| **定义** | 随队伍移动、**尚未入城区驻留** 的 **玩家人口**；**随行人数**，**不是** 编制 |
| **工作** | **不参与** 队伍工作；**不计入** `GetWorkableTeamHeadcount` |
| **载荷** | **默认人口的物品载荷**（占队伍 **载荷/背包容量**） |
| **粮食** | 计入 **队伍容器** 需求；同吃 **载荷粮食**；未分到 §2.4.5 半数减员 |
| **归属** | 接纳为随行人员时 **已属玩家** |
| **住宅** | 来源获取时 **预先规划安置**（OPEN-033） |
| **与运输队** | **运输载具类**（大物资载荷）与 **随行人员** **分轨** |

```text
TeamContainerDemand = RosterHeadcount + EscortAccompanyingCount
TeamCargoLoad       = MaterialCargo + EscortPopulationItemLoad
FoodDemand          = (Roster + Escort) × 1 粮食/人/周
```
