## 目标

支撑即时通讯、飞信同步、待同步队列与**关系事件**回传；参数全部走 SO，程序提供 `TryInstantComm`、`TryMessageSync` 等能力入口（见 [队伍与战斗数据结构](./队伍与战斗数据结构.md)）。

## 枚举定义

### `pending_sync_kind`

| 值 | 说明 |
|----|------|
| `vision` | 视野 / 地图情报（资源揭示、敌人位置等） |
| `relation_incident` | 关系事件（侵害行为，送达 `home_city_ref` 后更新城市关系） |

### `resource_reveal_level`

与 [单位类型与视野 · 资源点揭示等级](../../02-系统设计/06-单位与交战/单位类型与视野.md#资源点揭示等级) 一致：

| 值 | 说明 |
|----|------|
| `hidden` | **隐藏**：仅有资源点标记 |
| `kind` | **揭示种类** |
| `amount` | **揭示储量** |

### `vision_sync_status`

| 值 | 说明 |
|----|------|
| `pending` | 已发现，尚未同步到核心区 |
| `synced` | 已同步到玩家视野 |
| `expired` | 已失效（单位摧毁、超出保留规则等，规则待定） |

### `sync_channel`

| 值 | 说明 |
|----|------|
| `instant` | 即时通讯 |
| `message` | 飞信 |
| `core_zone` | 核心区即时视野（无需单位参与） |

## 同步流程（运行时）

与 [通讯与飞信（草稿） · 视野与同步流程](../../01-草稿/通讯与飞信.md#视野与同步流程) 对齐：

1. 单位发现新内容 → 写入 `vision_sync_pending`（`status=pending`）。
2. 若启用即时通讯且处于 `comm_station_config` 覆盖内 → 调用 `TryInstantComm` → 日志 `sync_channel=instant` → `status=synced`。
3. 否则保留在待同步队列；到达 `message_send_interval_turn` 或手动触发 → 生成飞信实例 → `TryMessageSync`。
4. 飞信到达核心区 → 批量更新 `pending` 为 `synced`，写入 `vision_sync_event_log`。
5. **`sync_kind=relation_incident`** 且已 `synced` → 调用关系结算，更新 `home_city_ref` 对行为方的城市关系（即时路径在写入时直接结算，不经过本队列，见 [势力系统](../../02-系统设计/05-城市与领袖/势力系统.md#单位归属来源城市与关系事件)）。
6. 受害单位歼灭且仍有未送达 `relation_incident` → 在 `death_hex` 生成 `relation_trace`（路径 A）。

## 单位状态与延迟宣告

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `death_sync_pending` | bool | 逻辑已死亡但尚未向玩家宣告；是否写入待同步队列 **待定**（OPEN-025） |
| `death_reveal_via` | string | 宣告来源：`vision` / `instant_comm` / `message` / 组合 **待定**（OPEN-025） |

- 逻辑死亡**不会**自动从玩家界面移除指令表/行动表；须等 [宣告死亡](../../02-系统设计/07-玩法循环/回合与行动表.md#死亡清理流程)（OPEN-026）。
- 飞信载荷是否包含「单位阵亡」事件、是否在到达前禁止玩家编辑该单位，见 OPEN-024、OPEN-025。
