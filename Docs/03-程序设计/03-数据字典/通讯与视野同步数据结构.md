## 目标

支撑即时通讯、飞讯同步与待同步队列；参数全部走 SO，程序提供 `TryInstantComm`、`TryMessageSync` 等能力入口（见 [队伍与战斗数据结构](./队伍与战斗数据结构.md)）。

## 枚举定义

### `vision_sync_status`

| 值 | 说明 |
|----|------|
| `pending` | 已发现，尚未同步到核心区 |
| `synced` | 已同步到玩家视野 |
| `expired` | 已失效（单位摧毁、超出保留规则等，规则待定） |

### `sync_channel`

| 值 | 说明 |
|----|------|
| `instant` | 即时通讯 |
| `message` | 飞讯 |
| `core_zone` | 核心区即时视野（无需单位参与） |

## 同步流程（运行时）

与 [通讯与飞讯系统 · 视野信息同步流程](../../02-系统设计/01-核心系统/通讯与飞讯系统.md#视野信息同步流程) 对齐：

1. 单位发现新内容 → 写入 `vision_sync_pending`（`status=pending`）。
2. 若启用即时通讯且处于 `comm_station_config` 覆盖内 → 调用 `TryInstantComm` → 日志 `sync_channel=instant` → `status=synced`。
3. 否则保留在待同步队列；到达 `message_send_interval_turn` 或手动触发 → 生成飞讯实例 → `TryMessageSync`。
4. 飞讯到达核心区 → 批量更新 `pending` 为 `synced`，写入 `vision_sync_event_log`。

## 单位状态与延迟宣告

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `death_sync_pending` | bool | 逻辑已死亡但尚未向玩家宣告；是否写入待同步队列 **待定**（OPEN-025） |
| `death_reveal_via` | string | 宣告来源：`vision` / `instant_comm` / `message` / 组合 **待定**（OPEN-025） |

- 逻辑死亡**不会**自动从玩家界面移除指令表/行动表；须等 [宣告死亡](../../02-系统设计/02-玩法循环/回合与行动表.md#死亡清理流程)（OPEN-026）。
- 飞讯载荷是否包含「单位阵亡」事件、是否在到达前禁止玩家编辑该单位，见 OPEN-024、OPEN-025。
