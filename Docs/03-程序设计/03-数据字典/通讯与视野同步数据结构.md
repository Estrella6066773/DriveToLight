## 目标

支撑即时通讯、待同步队列与**关系事件**回传；参数全部走 SO，程序提供 `TryInstantComm` 等能力入口（见 [队伍与战斗数据结构](./队伍与战斗数据结构.md)）。**飞信机制已废止**，下文 `message` 相关字段保留仅为历史占位。

## 枚举定义

### `pending_sync_kind`

| 值 | 说明 |
|----|------|
| `vision` | 视野 / 地图情报（资源揭示、敌人位置等） |
| `relation_incident` | 关系事件（人口损失；**单位轨**送达 `home_city_ref` 后更新**领袖关系**；**城区轨**见 [势力系统 · 分轨累计](../../02-系统设计/05-城市与领袖/势力系统.md#分轨累计已定)） |

### `resource_reveal_level`

与 [单位类型与视野 · 资源点揭示等级](../../02-系统设计/06-单位与交战/单位类型与视野.md#资源点揭示等级) 一致：

| 值 | 说明 |
|----|------|
| `hidden` | **隐藏**：仅有资源点标记 |
| `kind` | **揭示种类** |
| `amount` | **揭示储量** |

### `vision_sync_status`

| 值 | 说明 |
|----|------|
| `pending` | 已发现，尚未同步到核心区 |
| `synced` | 已同步到玩家视野 |
| `expired` | 已失效（单位摧毁、超出保留规则等，规则待定） |

| 值 | 说明 |

## 同步流程（运行时）

与 [通讯与视野系统](../../02-系统设计/06-单位与交战/通讯与视野系统.md) 对齐：

1. 单位发现新内容 → 写入 `vision_sync_pending`（`status=pending`）。
2. 若启用即时通讯且处于 `comm_station_config` 覆盖内 → 调用 `TryInstantComm` → 日志 `sync_channel=instant` → `status=synced`。
3. 否则保留在待同步队列，待覆盖恢复或回城等机制批量同步（细则 **待定**，OPEN-013）。
4. 批量同步成功 → 写入 `vision_sync_event_log`。
5. **`sync_kind=relation_incident`** 且已 `synced` → 调用关系结算，更新受害城**城市领袖**对行为方**城市领袖**的**领袖关系**（即时路径在写入时直接结算，不经过本队列，见 [势力系统](../../02-系统设计/05-城市与领袖/势力系统.md#人口损失与关系事件传导)）。
6. 受害**单位**歼灭且仍有未送达 `relation_incident` → 在 `death_hex` 生成 `relation_trace`（路径 A，**单位轨**）。

**情报延迟边界**：本表仅承载 **地图情报** 与 **关系事件** 的同步；己方队伍 `is_alive` 状态**不**经本队列延迟，见 [回合与行动表 · 队伍阵亡与清理](../../02-系统设计/07-玩法循环/回合与行动表.md#队伍阵亡与清理)。
