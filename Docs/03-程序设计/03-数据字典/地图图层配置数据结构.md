## 1. 设计目标

为数值策划提供**行业常见、可导入、可合成**的地图参数表系，对齐本项目的 GAS-lite（defs → GE/GA）与 hex ASC 运行时。

策划主表列名与能力说明口径见 [策划可读配置表规范](../../00-规范/策划可读配置表规范.md)。

**本文只讲各表作用与字段语义；具体数值见各 `*_defs.csv`。**

## 2. 核心原则

| 原则 | 说明 | 行业对照 |
|------|------|----------|
| **Defs 单表** | 每层一张 `L{n}_{layer}_defs.csv`，`{layer}_id` 为 PK | Unreal DataTable / Unity SO 一行一类 |
| **枚举集中** | `shared/enums.csv` 统一合法取值 | Schema registry |
| **合成显式** | `shared/modifier_stack.csv` 规定叠算顺序 | GameplayEffect 优先级栈 |
| **跨层最小化** | `L0_cross_modifiers.csv` 仅多实体交互 | 条件 Modifier，非重复 defs |
| **程序分离** | `_program/` 存放 GE/Tag 映射，策划不填玩法数 | 导入管线 / 代码生成 |
| **策划可读** | 主表中文列名 + [能力说明.csv](./tables/shared/能力说明.csv) | 列对照 [column_registry.csv](./tables/shared/column_registry.csv) |

**单位层**不在此体系：单位移动占格，不做格基线类型配置。

## 4. shared 表

### `enums.csv`

**表格作用说明**

全项目地图相关 CSV 的**枚举注册表**。凡字段标注 `enum:某组名`，合法取值必须来自本表对应 `enum_group` 下的 `enum_key` 行。策划新增类型时，若用到新枚举取值，须先在此登记，再在 defs 中引用；避免在单元格内自造未登记字符串。

**字段**

| 字段 | 说明 |
|------|------|
| `enum_group` | 枚举组名，如 `build_mode`、`mobile_city_pass` |
| `enum_key` | 该组内合法取值（写入 defs 单元格的值） |
| `display_name_zh` | 策划/UI 显示名 |
| `definition_zh` | 取值语义说明 |
| `sort_order` | 同组内排序（工具展示用，不影响逻辑） |

---

### `hex_stat_keys.csv`

**表格作用说明**

登记 hex 合成栈中可被读写的 **stat 键**及其适用图层。`L0_cross_modifiers.csv` 的 `stat_key`、程序侧 `HexStatResolver` 均以此表为权威键名表；避免跨表各写各的字段名。

**字段**

| 字段 | 说明 |
|------|------|
| `stat_key` | **PK**；程序 Attribute / 快照字段名 |
| `display_name_zh` | 可读名 |
| `value_type` | `int` / `bool` / `percent` / `enum:组名` 等 |
| `layer_scope` | 哪些图层可写入该 stat，如 `L1;L2;L5` |
| `notes_zh` | 备注 |

---

### `modifier_stack.csv`

**表格作用说明**

规定**格上有效参数**从底到顶的叠算顺序。运行时按 `stack_order` 从小到大依次应用各来源；后序来源在同名 stat 上覆盖或累加（由来源类型决定，见程序 `HexStatResolver`）。策划调平衡时须理解：改 L1 是改全局基线，改 L2 是在基线上加减，改 L5 `on_valid_*` 是设施有效时的临时覆盖。

**字段**

| 字段 | 说明 |
|------|------|
| `stack_order` | **PK**；数值越小越先结算 |
| `source_kind` | 来源种类，如 `layer_defs`、`facility_host`、`cross_modifier` |
| `source_ref` | 对应表文件名（无扩展名），如 `L1_terrain_defs`、`L0_cross_modifiers` |
| `description_zh` | 该步在玩法上的含义 |

默认栈序（行数据见 CSV）：

| `stack_order` | 含义 |
|:-------------:|------|
| 10 | L1 地形基线 |
| 20 | L2 环境修正 |
| 30 | L5 地形类设施宿主格覆盖 |
| 40 | defs 建造白名单列 |
| 50 | L0 跨层规则 |

## 5. 各层 defs 表

### `L1_terrain_defs.csv`

**表格作用说明**

地形层**唯一策划主表**。每行一种地形（平原、丘陵、山地等），描述该格在**无有效地形类设施实例**时的玩法基线：单位路径消耗、能否进入、可建造设施档位、移动城市通行性，以及允许例外建造的设施白名单。

地图生成时为每格写入 `terrain_id`（策划列 **类型编号**）并据此初始化 hex ASC；移动、建造、移动城市路径结算均以本表为**第一层**输入（再经 `modifier_stack` 后续步骤合成）。地形改造通过 **改造能力说明** 列与 [能力说明.csv](./tables/shared/能力说明.csv) 关联 GA，改造完成后可替换类型编号并重载本行。

不描述桥梁/通道建成后的覆盖——那属于 `L5_facility_defs.csv` 的 `on_valid_*` 列。

**字段（策划列名 · 程序键）**

| 策划列名 | 程序键 | 类型 | 说明 |
|----------|--------|------|------|
| **类型编号** | `terrain_id` | string | **PK** |
| **显示名称** | `display_name_zh` | string | 显示名 |
| **路径消耗（格）** | `path_cost_equiv` | int | 单位路径等效格数，≥1 |
| **单位可否进入** | `unit_enter_allowed` | bool | 填 **是** / **否** |
| **建造档位** | `build_mode` | enum:build_mode | 全部允许 / 仅辅助类 / 全部禁止 |
| **允许建造的设施** | `build_allow_facility_ids` | string | 分号分隔；**优先于**建造档位 |
| **移动城市通行** | `mobile_city_pass` | enum:mobile_city_pass | 可通过 / 可跨越 / 不可通过 |
| **移动城市路径消耗（格）** | `mobile_city_path_cost_equiv` | int | 可空；空 = 与路径消耗相同 |
| **改造能力说明** | `transform_ability_desc` | string | 可空；自然语言，详表见能力说明 |
| **程序标签** | `tags` | string | 分号分隔；授予 hex ASC 的 Tag |
| **状态** | `status` | enum:record_status | 已定 / 待定 |
| **备注** | `notes_zh` | string | 备注 |

列名对照见 [column_registry.csv](./tables/shared/column_registry.csv)。字段展开见 [地形类型配置数据结构](./地形类型配置数据结构.md)。

### `L2_environment_defs.csv`

**表格作用说明**

环境层策划主表。描述格上**表层状况**（树林、焦土、弹坑）与**区域/大气效果**（沙暴、暴雨、迷雾、移动城市轨迹）对玩法的修正。与 L1 正交：同一格可同时有丘陵（地形）与密林（环境）。

本表不定义地形形状，只提供对路径、通行、视野、建造档位等的**增量或覆盖**（如 `path_cost_delta`）。`stack_rule` 决定同格多个环境实例能否共存。存续 `duration_turns` 为空表示直至被移除。

**字段**

| 字段 | 说明 |
|------|------|
| `environment_id` | **PK** |
| `display_name_zh` | 显示名 |
| `subtype` | enum:environment_subtype；地面表层 / 天气 / 城市轨迹 |
| `stack_rule` | enum:environment_stack；互斥 / 可叠 / 区域 |
| `path_cost_delta` | 对 `path_cost_equiv` 的加减；空 = 0 |
| `unit_enter_override` | 可空；覆盖 L1 的 `unit_enter_allowed` |
| `vision_radius_pct` | 单位视野修正 %；空 = 0 |
| `watchtower_vision_bonus` | 格上侦察塔视野加成（int，加算）；L0_cross 叠加器，密林−1 / 丘陵+1 / 山地+2；与侦察塔 base +3 加算 |
| `duration_turns` | 默认存续回合；空 = 直至移除 |
| `build_mode_override` | 可空；覆盖合成后的建造档位 |
| `tags` | 分号分隔 Tag |
| `status` | enum:record_status |
| `notes_zh` | 备注 |

---

### `L3_resource_defs.csv`

**表格作用说明**

资源层策划主表。每行一种**资源点类型**（村镇、矿藏、果地、遗迹），定义储量产出种类、默认揭示等级、储量区间，以及可在该资源点上建造的**采集设施白名单**。

资源点与地形正交：沼泽上若有矿藏，仍可通过本表 `build_allow_facility_ids` 覆盖 L1 的 `build_mode=none`。`rebuildable=false` 表示资源点本体不可重建（遗迹）。具体格上储量数值在关卡实例数据中配置，本表只定义**类型级**规则。

**字段**

| 字段 | 说明 |
|------|------|
| `resource_id` | **PK** |
| `display_name_zh` | 显示名 |
| `reserve_kind` | 储量产出种类：`metal` / `food` / `energy` / `population` |
| `build_allow_facility_ids` | 分号分隔；允许建造的 `facility_id`，优先于地形 `build_mode` |
| `rebuildable` | 资源点本体可否重建 |
| `reveal_default` | enum:resource_reveal；生成时默认揭示等级 |
| `reserve_min` / `reserve_max` | 刷新储量区间；可空或 `pending` |
| `extract_work_type` | 开采绑定的工作类型 id |
| `status` | enum:record_status |
| `notes_zh` | 备注 |

---

### `L4_district_defs.csv`

**表格作用说明**

建筑层（城区）策划主表。每行一种**城区类型**（核心区、一般城区、城坞等），定义居民承载、航行负载成本、设施建造位数、废墟阈值等**类型级**参数。

建筑层每格至多一个城区实例；**状态**（正常/废墟）由运行时实例持有，本表通过 `ruins_threshold_pct`、`ruins_allows_work` 等提供类型默认值。特殊城区与一般城区的分轨见 `category` 枚举。

**字段**

| 字段 | 说明 |
|------|------|
| `district_id` | **PK** |
| `display_name_zh` | 显示名 |
| `category` | enum:district_category；`special` / `general` |
| `base_resident_cap` | 基础居民承载 |
| `load_step_grids` | 负载成本：每航行 x 格累积一步 |
| `load_repair_cost` | 负载成本：每步等价修复资源量 y |
| `load_repair_kind` | 修复资源类型，如 `metal` |
| `facility_slots` | 一般城区设施建造位数；特殊城区常为 0 |
| `routine_cost_json` | **日常开销**：工作区开启且人口满足期间的周期资源扣减（JSON，如 `{"energy":10}`）；**无被动能力**或未配置开销时**留空** → 视为无开销 |
| `activation_cost_json` | **激活开销**：玩家**激活主动能力**时扣减的资源（JSON）；**无主动能力**或未配置时**留空** → 视为无开销 |
| `ruins_threshold_pct` | 结构完整度低于此值转废墟 |
| `ruins_allows_work` | 废墟状态下是否允许工作 |
| `status` | enum:record_status |
| `notes_zh` | 备注 |

### `L5_facility_defs.csv`

**表格作用说明**

设施层策划主表。每行一种**设施类型**，统一描述建造/运维参数、占格分类，以及**地形类设施**在宿主格上有效时的覆盖行为（`on_valid_*` 列）。

占格类生产/仓储/屋舍与辅助类（哨塔、城墙等）共用本表，通过 `category`、`subtype` 区分。桥梁、通道的通行与建造覆盖是设施**固有行为**，写在 `subtype=terrain` 行的 `on_valid_*` 中，不写入 L1 或独立 Buff 表。击毁时若 `revert_on_destroy=true`，宿主格回退至 L1 基线（及已合成的 L2 等）。

**字段**

| 字段 | 说明 |
|------|------|
| `facility_id` | **PK** |
| `display_name_zh` | 显示名 |
| `category` | enum:facility_category；`occupying` / `auxiliary` |
| `subtype` | enum:facility_subtype；`terrain` / `production` / `storage` / `dwelling` |
| `production_tier` | enum:production_tier；生产类专用；可空 |
| `build_work_type` | 建造工作类型 id |
| `build_work_amount` | 建造工作量（回合数） |
| `durability_max` | 耐久上限 |
| `maintain_work_type` | 运维工作类型 id |
| `storage_bonus` | 仓储类容量加成 |
| `resident_cap_bonus` | 屋舍额外居民承载 |
| `pop_loss_reduction_pct` | 城墙等：本格人口损失减免 % |
| `host_terrain_ids` | 地形类：可建宿主地形，分号分隔 |
| `on_valid_path_cost` | 地形类：有效时覆盖路径等效 |
| `on_valid_build_mode` | 地形类：有效时覆盖建造档位 |
| `on_valid_unit_enter` | 地形类：有效时覆盖单位可否进入 |
| `on_valid_span_connect` | 地形类：有效时启用跨阻断连通 |
| `revert_on_destroy` | 地形类：击毁后是否回退宿主格 |
| `status` | enum:record_status |
| `notes_zh` | 备注 |

### `L6_item_defs.csv`

**表格作用说明**

物品层策划主表（**已中文化列名**，对照见 [column_registry.csv](./tables/shared/column_registry.csv)）。每行一种**可落于格上或被拾取的物品类型**，定义物品类型、金币基准价、堆叠上限、搬运重量、是否必须由队伍拾取等。设施产出、资源点开采、城区仓储等与物品层的交互，通过物品 id 引用本表，不在此重复设施逻辑。

**字段**

| 字段 | 说明 |
|------|------|
| `item_id` | **PK**（列名「类型编号」） |
| `display_name_zh` | 显示名 |
| `category` | enum:item_category（列名「物品类型」） |
| `gold_base_price` | **金币基准价**；金币为纯计价单位（无实物、无专属数据字段），本列是贸易**静态基准价表**的取值来源；计价锚：**1 份能源 = 50 金币**；可空 = 待定（见 [领袖与势力 · 领袖贸易](../../02-系统设计/05-城市与领袖/领袖与势力.md#领袖贸易已定)） |
| `stack_hex_max` | 单格最大堆叠 |
| `stack_warehouse_max` | 仓库最大堆叠 |
| `weight_per_unit` | 单位搬运重量 |
| `pickup_requires_team` | 是否须队伍拾取 |
| `status` | enum:record_status |
| `notes_zh` | 备注 |

### `L0_cross_modifiers.csv`

**表格作用说明**

**跨图层** stat 修正表。仅收录必须同时满足「某层某类型 + 另一层某实体」才生效的规则（如环境为密林 **且** 格上有哨塔 → 削弱视野）。在 `modifier_stack` 中处于第 50 步，按 `priority` 从小到大在同步内排序。

**禁止**写入单层 defs 已能表达的内容（如河流 `build_mode=none`、通道路径覆盖）——那些分别由 L1、L5 负责，避免双处维护。

**字段**

| 字段 | 说明 |
|------|------|
| `modifier_id` | **PK** |
| `priority` | 同 stack 50 内优先级；越小越先 |
| `when_layer` | 触发层：`terrain` / `environment` / `facility` 等 |
| `when_ids` | 触发类型 id，分号分隔 |
| `require_layer` | 并存条件层；可空 |
| `require_ids` | 并存条件类型 id，分号分隔 |
| `stat_key` | → `hex_stat_keys.stat_key` |
| `op` | `set` / `add` / `multiply` |
| `value` | 修正值 |
| `status` | enum:record_status |
| `notes_zh` | 备注 |

## 6. _program 表（策划只读）

### `build_grant_tags.csv`

**表格作用说明**

将 defs 中的 `build_allow_facility_ids` 映射为运行时 `Allow.Build.{facility_id}` Tag，供 `TryBuild` 与导入管线使用。策划**只维护 defs 白名单列**；本表由程序/工具根据设施 id 同步，一般不需手改玩法数值。

**字段**

| 字段 | 说明 |
|------|------|
| `facility_id` | **PK**；FK → `facility_defs` |
| `grant_tag` | 运行时 Tag，如 `Allow.Build.bridge` |
| `rule_priority` | 与 `build_mode` 冲突时的优先级 |
| `notes_zh` | 备注 |

---

### `ge_catalog.csv`

**表格作用说明**

defs 列值 → `GameplayEffect` / `GameplayAbility` 的**导入映射目录**。程序根据 `derived_from` 条件在加载 `terrain_defs` 等表时自动 Apply 对应 GE/GA。策划调平衡时**改 defs 数值即可**，不应在本表填写路径、建造档位等玩法数。

**字段**

| 字段 | 说明 |
|------|------|
| `capability_id` | **PK**；对应 SO 的 effect/ability id |
| `capability_kind` | `gameplay_effect` / `gameplay_ability` |
| `executor_key` | GA 程序入口；GE 可空 |
| `attribute_id` | GE 修正的 Attribute |
| `modifier_op` | `set` / `add` / `multiply` |
| `granted_tags_json` | 授予 Tag |
| `blocked_tags_json` | 阻断 Tag |
| `required_tags_json` | GA 所需 Tag |
| `impl_status` | `defined` / `pending` |
| `derived_from` | 触发条件，如 `L1_terrain_defs.build_mode=none` |
| `notes_zh` | 备注 |

## 7. 建造结算（TryBuild）

1. `build_allow_facility_ids`（L1/L3）→ `_program/build_grant_tags` → `Allow.Build.*`
2. 否则按合成后 `build_mode`
3. `facility_defs` 几何与放置规则 + 设计文档 `rule_guard`

## 8. 程序映射（总览）

| 输入 | 运行时 |
|------|--------|
| L1～L6 `L{n}_*_defs.csv` | 导入各层 `*ConfigSO` |
| `shared/modifier_stack.csv` | `HexStatResolver` 叠算顺序 |
| `_program/ge_catalog.csv` | `GameplayEffectConfigSO` / GA |
| `_program/build_grant_tags.csv` | hex ASC 建造 Tag |
