## 职责边界

- **负责**：读取 SO 配置、校验能力通道、执行指令队列条目、写入战斗日志、更新队伍实例状态。
- **不负责**：行动表排序（见 [回合与行动数据结构](../03-数据字典/回合与行动数据结构.md)）、视野同步（见 [通讯与视野同步数据结构](../03-数据字典/通讯与视野同步数据结构.md)）、关系数值变更规则（见关系系统，待补数据表）。

## 概念区分

| 概念 | 数据对象 | 作用 |
|------|----------|------|
| **指令表** | `team_command_queue` + `team_command_item` | AI **任务表**：跨回合任务规划；按 `order_index` 逐步推进 |
| **行动表** | `action_table_entry` | 本回合内单位行动先后顺序；见 [回合与行动数据结构](../03-数据字典/回合与行动数据结构.md) |

同一回合内：行动表决定**轮到哪个单位**；轮到后，单位从指令表 `current_index` 条目出发，由 **AI 自主决定**本回合如何推进该任务（路径、遇敌应对等），而非玩家逐步微操。

设计规则见 [回合与行动表 · 指令表与自主执行](../../02-系统设计/02-玩法循环/回合与行动表.md#指令表与自主执行)。

## 单条指令执行管线

在**玩家行动**或 **AI 行动**阶段，某队伍轮到行动时，按以下顺序判定：

1. 读取 `team_command_queue.current_index` 对应 `team_command_item`。
2. 若条目 `status` 为 `done` / `skipped`，跳至下一条。
3. 校验 `required_ability_channel` 是否在模板能力集合中且 `enabled=true`；否则 `fail_reason=ability_not_enabled`。
4. 读取 `ability_channel_config.rule_guard_json` 与 `target_filter_json`，校验路径、关系、行动点等。
5. 读取对应 `ability_param_*` 与 `headcount_factor_curve`，计算运行时参数写入 `ability_runtime_json`。
6. 若当前条目为工作中类任务且 `work_duration_locked=false`：调用 `ResolveWorkEfficiency`；若 `work_subject_ref` 上已有 `work_subject_state` 且类型匹配，继承完成度与锁定时长。
7. 若 `status=running` 且本回合无中断：调用 `AdvanceWorkProgress`；满完成度则 `TryCompleteWork`。
8. 若遭遇意外 / 行动打断：调用 `SuspendWork`（保留完成度），完成意外结算后 `ResumeWork` 或下回合继续。
9. 调用 `executor_key` 对应入口（移动、交战等并行逻辑）。
10. 若本回合未完成整条指令，保持 `status=running` / `suspended`；完成则 `done`；失败则按重试规则处理。

### 失败与重试

- `retry_count < max_retry`：可在同一 `target_ref` 内调整路径或攻击点后重试，不递增 `current_index`。
- 重试耗尽：写 `report_text`，`status=failed`，`current_index++`，进入下一条。
- 持续受阻且 AI 策略为保守：可写入跳过说明并 `status=skipped`（仍递增索引）。

## 工作效率结算（`ResolveWorkEfficiency`）

工作开始时（管线第 6 步），按 [回合与行动数据结构 · 工作效率与时长结算](../03-数据字典/回合与行动数据结构.md#工作效率与时长结算)：

1. 读取 `work_type_config.base_duration_turn` → `work_base_duration_turn`。
2. 若 `apply_headcount_factor`：追加乘法项 `headcount_ratio = current_headcount / default_headcount` 至 `work_efficiency` 通道。
3. 调用各 **`stat_modifier_provider`**（城模块、地形、状态等，SO 注册）追加加法 / 乘法项；每项须含 `source_id` 与 `source_display_name`。
4. 合成 `work_efficiency_final`，计算 `work_duration_turn = max(1, ceil(base / efficiency))`。
5. 写入 `stat_value_channel_entry` 与 `work_efficiency_snapshot`；UI 只读展示。

**锁定规则**：结算完成后置 `work_duration_locked=true`。当前条目进行中，人数或模块修正变化**不**触发重算；**下一项**工作类指令开始时再次调用本函数（`work_duration_locked=false`）。

指挥阶段预览须调用**同一函数**（传入预览上下文），保证玩家所见与执行一致。

## 工作推进与中断

与 [回合与行动表 · 完成度与推进](../../02-系统设计/02-玩法循环/回合与行动表.md#完成度与推进)、[工作中断与恢复](../../02-系统设计/02-玩法循环/回合与行动表.md#工作中断与恢复) 对齐。

| 入口 | 行为 |
|------|------|
| `ResolveWorkEfficiency` | 工作**首次开始**时锁定时长；生成数值通道 |
| `AdvanceWorkProgress` | `work_progress_turn += 1`；可选同步 `work_subject_state` |
| `SuspendWork` | 读取 `work_type_config.allow_suspend`；`true` 则 `status=suspended` 且**不**改完成度 |
| `ResumeWork` | `status=running` |
| `TryCompleteWork` | 完成度满 → 对**被工作对象**施加效果 → `done` |

**中断来源**

| 来源 | 处理 |
|------|------|
| 意外（地图响应、遇敌等） | 本回合 `SuspendWork` → 结算 → `ResumeWork` 或下回合继续 |
| 行动打断（交战占满行动点、强制位移等） | 同上 |
| 主动中止（指挥阶段改指令） | 读 SO **`allow_suspend`**；`true` 保留完成度 |

**恢复同一被工作对象**：`work_subject_ref` + `work_type_id` 与 `work_subject_state` 一致时，读取已存进度；`work_actor_ref` 可不同（换队续作待定，OPEN-032）。

## 遇敌判定与策略分支

在单条指令执行管线第 4 步（路径 / 关系校验）之前或之中，若检测到敌对单位构成威胁（进入威胁范围、阻断路径、或被动卷入交战），读取 `team_instance_state.ai_strategy`，按 [队伍系统 · 遭遇敌人时的行为](../../02-系统设计/01-核心系统/队伍系统.md#遭遇敌人时的行为) 分支：

| `ai_strategy` | 程序侧优先级 | 允许的调整动作（不改动任务目标类型） |
|---------------|--------------|--------------------------------------|
| `conservative` | 降低损失 → 推进任务 | `TryReroute`（改道）、`TryDisengage`（脱离接触）、`TryWait`（等待）；无武装时不调用 `TryAttack` |
| `aggressive` | 推进任务 → 损失控制 | 优先继续当前 `command_type` 与 `target_ref`；已启用 `attack_strike` 时可 `TryAttack`；否则仍用改道 / 等待 / 穿越 |

实现要点：

- 分支逻辑**只读** `ai_strategy` 与能力 SO（是否启用 `attack_strike`、是否 `is_armed`），不在代码中写死队伍类型。
- 无武装队伍在 `aggressive` 下也不得调用 `TryAttack`；仅通过非战斗手段推进任务。
- 同一 `team_command_item` 内调整失败且达到 `max_retry` 后，写 `report_text` 并进入下一条；保守策略可额外标记 `skipped`。

## 意外中断与恢复

与 [回合与行动表 · 遭遇意外的中断与恢复](../../02-系统设计/02-玩法循环/回合与行动表.md#遭遇意外的中断与恢复) 对齐：

1. **检测意外**：响应（陷阱等）、遇敌、环境效果、行动点不足等，在当前条目自主执行过程中触发。
2. **中断**：完成意外相关结算；**不**修改 `team_command_queue.current_index`；**不**重置 `work_progress_turn`；调用 `SuspendWork` 或保持 `running` 待本回合恢复。
3. **恢复**：
   - 本回合 `action_point` 仍有剩余且 `rule_guard` 允许 → `ResumeWork` 后在同一 `command_id` 内继续；
   - 否则 → 本回合结束；下一回合从**同一完成度**继续。
4. **终止条目**：仅当重试耗尽或 AI 接管规则判定「持续受阻且无法推进」时，`current_index++`。

意外类型与地图 [响应](../../02-系统设计/01-核心系统/地图图层.md#响应) 的具体触发条件仍由 OPEN-015 补全；本管线的**结构规则**（不打断指令表、恢复同一条目）已定。

## 战斗结算管线

在攻击类指令或反击触发时，按 [队伍系统 · 战斗结算流程](../../02-系统设计/01-核心系统/队伍系统.md#战斗结算流程当前版本) 顺序执行：

1. **目标校验**：`target_filter` + `faction_relation_state.is_hostile`（主动攻击时）。
2. **读取攻击参数**：`ability_param_attack` × 士气 × 人数系数 → `attacker_power_final`。
3. **读取防御参数**：防守方 `base_defense_power` × 士气 × 人数系数 → `defender_defense_final`。
4. **减员结算**：按攻防差计算 `defender_headcount_loss`（公式待 [04-数值框架](../../02-系统设计/04-数值框架/) 补全）。
5. **士气变化**：攻击方提升、受击方下降（具体数值待定）。
6. **写入** `combat_event_log`（`attack_kind=normal`）。
7. **反击判定**：若防守方启用 `counterattack_response` 且攻击者在 `counterattack_range_max` 内，执行一次反击流程，写入日志且 `counterattack_triggered=true`；反击条目 `attack_kind=counterattack`，**不得**再次触发反击。

## 与回合阶段的协作

| 回合阶段 | 本模块行为 |
|----------|------------|
| 玩家指挥 | 仅修改指令队列与行动表；不执行能力 |
| 玩家行动 | 按行动表顺序执行玩家阵营队伍指令 |
| AI 行动 | 外部城市按 `game_seed` 伪随机顺序行动；见 [回合与行动数据结构](../03-数据字典/回合与行动数据结构.md#外部城市行动顺序) |
| 环境行动 | 本模块不主动推进队伍指令；环境效果可打断 `running` 条目（规则见地图图层，待定） |

`combat_event_log.phase` 使用 [回合与行动数据结构](../03-数据字典/回合与行动数据结构.md) 中的 `turn_phase` 枚举。
