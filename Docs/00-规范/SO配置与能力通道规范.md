# SO 配置与能力通道规范

## 目标

统一项目实现口径：**参数全部走 ScriptableObject（SO）配置，程序只负责能力通道与执行流程**。

本规范适用于 `Docs/02-系统设计/` 与 `Docs/03-程序设计/` 的所有新增与修订文档。

## 核心原则

1. **参数配置化**
 - 队伍、单位、能力、战斗、建造、搬运、勘探等业务参数，必须通过 SO 配置。
 - 禁止在程序中写死平衡参数、默认值与阈值（仅允许编辑器兜底值与空值保护）。

2. **能力通道化**
 - 程序只提供能力执行入口（例如 `TryBuild`、`TryTransport`、`TryAttack`）。
 - 能力是否启用、参数如何取值、目标过滤与前置条件，统一由 SO 控制。

3. **模板组合化**
 - 队伍种类视为模板预设，本质为“能力集合 + 参数配置”。
 - 模板可复用同一能力通道，不同模板通过不同参数表现差异。

4. **引擎内配置**
 - 默认由开发者在 Unity 编辑器内完成 SO 资产配置与装配。
 - 可使用 MCP 代为批量创建、挂载、校验 SO 资产，但不改变“配置权在 SO”的原则。

5. **数值通道**
 - 合成类数值（如 **工作效率** → 工作时长）须通过**数值通道**记录每一项修正的**来源**、**运算方式**与**数值**。
 - 运行时写入 `stat_value_channel_entry`（及快照表如 `work_efficiency_snapshot`），**指挥预览与行动执行共用同一结算管线**。
 - 玩家界面须能罗列并展示通道明细；禁止仅在 UI 层硬编码说明文字而不落运行时记录。

## 禁止事项

- 在运行时代码中用 `if (队伍类型 == X) then 固定能力` 写死玩法能力。
- 在代码常量中固化可调平衡参数（攻击效果、负载上限、建造速度、视野等）。
- 在文档中只写“程序计算”而不提供对应 SO 字段来源。

## 必备设计要素

每个可执行能力在文档中至少要有以下字段：

- `enabled`（是否启用）
- `executor_key`（程序入口键）
- `param_profile_id`（参数配置引用）
- `target_filter`（目标过滤）
- `rule_guard`（前置条件）
- `cooldown` 与 `action_point_cost`（若该能力需要）

## 文档落地要求

1. 在系统设计文档中，说明能力通道与模板组合关系，不写死“类型=能力”。
2. 在程序设计文档中，给出 SO 表结构、字段约束、运行时读取流程。
3. 在数据字典中，给出至少一份可执行示例（模板、能力集合、能力参数）。
