# 文档库使用说明

## 1. 文档三分库

| 库 | 路径 | 回答的问题 |
|----|------|------------|
| **对外展示** | [游戏介绍.md](../游戏介绍.md) | 游戏是什么、机制概览、专题文档入口 |
| 系统设计 | [Docs/02-系统设计/](../02-系统设计/) | 玩家体验到什么、规则**理应**怎样 |
| 程序设计 | [Docs/03-程序设计/](../03-程序设计/) | 用什么表、模块、数据**具体**实现 |
| 设定 | [Docs/04-设定/](../04-设定/) | 世界与故事是什么 |

## 2. 文档元数据

每篇正式文档的文首（blockquote 或元数据表）须标注 **`状态`** 与 **`校验状态`**。二者分工不同，勿混用。

### 状态（写作进度）

| 取值 | 含义 |
|------|------|
| **草稿** | 想法未收敛，可大幅修改 |
| **评审中** | 基本写完，等待团队确认 |
| **已锁定** | 已通过评审，后续修改须走修订记录 |

### 校验状态（对照进度）

记录本文与**关联文档**、**成对实现**、**待细化追踪**之间是否已核对、是否存在已知偏差。

| 取值 | 含义 | 典型动作 |
|------|------|----------|
| **未校验** | 新建或近期大改，尚未与关联文档对照 | 默认取值；核对「相关系统 / 相关设定 / 成对文档」 |
| **已对照** | 已与关联文档核对，当前无已知 open 项 | 可配合 `状态：评审中` 或 `已锁定` |
| **部分符合** | 大体一致，但本文或关联文档仍有未关闭的待确认 / 追踪项 | 在「待确认事项」或 [待细化追踪](./待细化追踪.md) 留痕 |
| **待对齐** | 已知与关联文档或实现存在需修订的偏差 | 须在 [待细化追踪](./待细化追踪.md) 登记，并链回本文 |
| **不适用** | 规范说明、索引、模板、归档、草稿碎片等 | 用于 `00-规范/`、`README`、`_templates/`、`99-备份与归档/` 等 |

约束：

- `状态` 为 **已锁定** 时，`校验状态` 不得为 **未校验**。
- `校验状态` 为 **待对齐** 时，须在 [待细化追踪](./待细化追踪.md) 有对应 open 项（或本文「待确认事项」已写明偏差）。
- 元数据表写法示例：`| 校验状态 | 未校验 |`；blockquote 写法：`> 校验状态：未校验`。

## 3. 命名规范

- 优先使用 `前缀-标题.md`，例如：
  - `核心循环.md`
  - `能力系统-设计.md`
  - `能力系统-实现.md`
  - `数据表-PlayerState.md`
- 避免使用 `新建文本文档.md`、`未命名.md` 等无意义名称。

## 4. 成对维护

同一主题的设计与实现，用**可点击的双向链接**互相引用。

示例：

```markdown
<!-- 在 02-系统设计/01-核心系统/队伍系统.md 顶部 -->
> 实现文档：[模块划分](../03-程序设计/01-架构总览/模块划分.md)

<!-- 在 03-程序设计/01-架构总览/模块划分.md 顶部 -->
> 设计文档：[队伍系统](../02-系统设计/01-核心系统/队伍系统.md)
```

## 5. 草稿流程

1. 灵感、参考图、情绪板先扔进 `01-草稿/`。
2. 经过讨论收敛后，迁移到 `02-系统设计/` 或 `04-设定/`。
3. 进入开发阶段后，在 `03-程序设计/` 写实现文档。
4. 废弃草稿移动到 `99-备份与归档/`。

## 6. 正文交叉链接（对外展示文档）

对外展示入口为 [游戏介绍.md](../游戏介绍.md)。在 [游戏介绍.md](../游戏介绍.md)、[02-系统设计](../02-系统设计/)、[04-设定](../04-设定/) 等**对外展示**文档的正文中，**首次提及**下列概念时，应创建可点击链接指向对应文件：

- 项目内模块或文档标题（如「[队伍系统](../02-系统设计/01-核心系统/队伍系统.md)」「[四种核心资源](../02-系统设计/02-资源循环/四种核心资源.md)」）
- 已在文档库中单独立篇的专有名词（如「[移动城市](../02-系统设计/03-模块与城市/城市模块化.md)」「[荒野地点](../02-系统设计/02-资源循环/荒野地点.md)」）

规则：

- 同一文档内，链向同一目标仅链接**第一次**出现；后续重复提及保持纯文本。
- 文首元数据表中的「相关系统」「相关设定」仍应完整列出，不受「仅首次」限制。
- 链接使用相对路径；仅在目标文档有明确章节标题时使用锚点。
- [03-程序设计](../03-程序设计/) 等面向实现的文档以字段与模块键为主，不要求此规则。
- **太阳 / 人造太阳**：对外文档与玩家可见文本统一称「太阳」；「人造太阳」等人造本质仅写在 [05-隐秘真相/太阳的真相.md](../04-设定/05-隐秘真相/太阳的真相.md) 等隐秘真相条目内，**不得**出现在对外展示文档。

## 8. 项目级实现约束

全项目统一执行 [SO 配置与能力通道规范](./SO配置与能力通道规范.md)：

- 参数全部通过 SO 配置；
- 程序只负责能力通道与执行流程；
- 队伍种类按模板与能力组合实现，不以硬编码分支实现玩法差异。
